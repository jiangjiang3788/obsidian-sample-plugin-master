正文

可以，下面是 《统一输入修改计划表 v2（已纳入副作用管理）》。 我按你当前代码结构来排，已经把这些实际位置考虑进去了：

  • app/ui/components/QuickInputEditor/*
  • features/quickinput/*
  • features/settings/*ViewEditor.tsx
  • features/settings/InputSettings.tsx
  • app/usecases/*
  • app/store/slices/*
  • core/services/InputService.ts
  • core/utils/inputTemplateUtils.ts
  • core/utils/normalize.ts
  • core/types/quickInput.ts

先给一句总原则：

不要做第二个前台输入渠道。 要做的是：新建统一输入内核，让 quickinput 先接进去。

一、目标先说清

这次改造不是“优化 quickinput”，而是统一这 4 件事：

  1. 所有新建 / 修改 / 删除入口
  2. 所有字段缺失 / 类型缺失 / 主题缺失处理
  3. 所有 view / editor / modal 里的编辑动作
  4. 所有写入后的副作用和竞态控制

二、统一输入修改计划表 v2

阶段 要做什么 你要得到什么 主要看哪些地方 用 Think 还是 Pro
1 定边界 明确 features / app/usecases / core / platform / shared 各管什么 app/ARCH_CONSTRAINTS.md、app/usecases/*、features/* Think
2 盘点所有写入口 找全创建 / 编辑 / 删除 / 快速输入 / modal 保存入口 features/settings/*、features/quickinput/*、app/ui/components/QuickInputEditor/*、platform/modals/* Pro
3 盘点所有 usecase 找全现在系统里已有的标准动作 app/usecases/blocks.usecase.ts、layout.usecase.ts、settings.usecase.ts、theme.usecase.ts、timer.usecase.ts、viewinstance.usecase.ts Pro
4 找绕过点 找出哪些地方直接改 store / slice / repository features/settings/*、QuickInputEditor/*、features/quickinput/*、app/store/* Pro
5 盘点校验与补全 找全字段缺失、类型缺失、主题缺失、默认值处理逻辑 core/services/InputService.ts、core/utils/inputTemplateUtils.ts、core/utils/normalize.ts、features/settings/InputSettings.tsx Pro
6 定统一动作模型 把所有入口收束成少数标准动作 基于前面盘点结果 Think
7 设计统一输入内核 明确新目录、新 usecase、新 validation 放哪 app/usecases/、core/services/、core/types/ Think
8 盘点副作用 找出 create/update/delete 后现在会触发什么 view 刷新、toast、modal、layout、事件通知、平台同步 Pro
9 定副作用归属 决定哪些副作用放 UI，哪些放 usecase，哪些放服务层 结合 2~8 步结果 Think
10 定竞态与防重规则 统一处理重复提交、并发更新、删除冲突 提交逻辑、modal、quickinput、view editor Think
11 先接 quickinput 到统一内核 不改前台入口，先改背后动作链 features/quickinput/*、QuickInputEditor/*、InputService.ts Pro
12 试点迁移 1~2 个 view 编辑入口 验证方案可行，不一次性全改 优先选最常用 view/editor Pro
13 全量迁移其余编辑入口 所有写入口最后都走统一动作 所有 view / editor / modal / settings 入口 Pro
14 清理旧逻辑 删除重复路径和临时兼容层 quickinput 旧逻辑、散落的补全逻辑、绕过点 Think
15 验收 确认没有双渠道、没有绕过点、没有副作用乱飞 全局检查 Think

三、你现在最该先做的 6 项

优先级 事项 原因 用 Think 还是 Pro
1 收集所有 UI 写入口 不先找全，后面一定漏 Pro
2 收集所有 usecase 你要统一的是动作，不是按钮 Pro
3 收集所有缺失处理和补全逻辑 规则现在很可能分散 Pro
4 定统一动作名 不先统一动作名，后面会乱 Think
5 定副作用归属 不然统一入口后副作用还是散 Think
6 定 quickinput 的新角色 这是避免双渠道的关键 Think

四、quickinput 这次到底怎么处理

不建议的两种做法

方案 为什么不建议
继续直接把 quickinput 做大 会越来越像“输入系统本体”，名字和职责失真
再做一个新输入前台入口 真的会变成双渠道

建议的做法

做法 说明
保留 quickinput 作为前台入口 用户入口不变
新建统一输入内核 真正的 create / update / delete / validate / normalize 放进去
让 quickinput 先接统一内核 先做一条主线
后面再让 view / modal / settings 入口逐步接入 最后所有入口走同一套动作

一句话

quickinput 留着，但降级成入口壳。

五、推荐的统一动作模型

你先不要做太大，先收成这些标准动作：

动作名 作用
openCreateRecord 打开新建入口
openEditRecord 打开编辑入口
submitCreateRecord 提交新建
submitUpdateRecord 提交修改
submitDeleteRecord 提交删除
validateRecordInput 校验字段 / 类型 / 主题等是否合法
normalizeRecordInput 统一补默认值和缺失项
resolveMissingDependencies 统一处理字段缺失 / 类型缺失 / 主题缺失

重点是：

UI 只负责触发这些动作。 UI 不自己决定怎么保存、怎么补齐、怎么刷新。

六、副作用管理这次要怎么纳入

这次不要另开一整套大工程，但一定要纳入主计划。

你至少要纳入这 4 类副作用

副作用类型 要不要这次纳入
数据写入（store / repository） 要
UI 反馈（toast / modal / panel） 要
视图刷新（view / layout / widget refresh） 要
防重与竞态（重复提交 / 覆盖更新 / 删除冲突） 要
埋点 / 日志 可以后放
性能优化 可以后放

推荐归属

逻辑 放哪
点击按钮 / 打开弹窗 UI 层
create / update / delete 主动作 app/usecases
validate / normalize core 的输入服务或 validation 层
写 store / repository store / repository
toast / refresh / close modal 尽量由 usecase 统一触发或由调用方按统一结果处理

七、竞态和防重，这次顺便定掉

这次不一定一次修完，但规则必须先定。

问题 最低处理标准
连点两次提交 提交中锁按钮
同一 record 被多个入口同时编辑 同 recordId 串行提交
删除和更新同时发生 更新前检查对象仍存在
缺失补全覆盖用户输入 normalize 只在提交前做一次
quickinput 和 view editor 同时改一条记录 统一走同一个 update usecase
modal 保存后 view 没刷新 提交成功后统一刷新路径

八、你需要补收集的东西

除了“所有 UI 点击编辑修改部分”，这次还要补这几类：

需要补收集 为什么
所有新建入口 create 主线必须统一
所有删除入口 删除最容易和更新冲突
所有 inline edit 最容易绕过统一入口
所有 modal 保存按钮 常带特殊逻辑
所有直接写 slice / repository 的地方 这是最大绕过点
所有默认值补全逻辑 可能多处重复执行
所有保存后刷新视图的地方 这是副作用分散的高发区

九、什么标准算“可以开始改”

你满足下面这些，就可以正式动手：

标准 达到说明
写入口找到了 80% 以上 不会迁一半才发现漏入口
usecase 覆盖 create / update / delete 主线 动作已经成型
字段缺失 / 类型缺失 / 主题缺失逻辑已找全 校验规则能收口
quickinput 新角色明确 不会继续做胖
至少选定 1 个试点 view/editor 可以小步验证
副作用归属已定 迁移后不会副作用更乱
竞态规则已定 不会统一后更容易炸

十、推荐的实际执行顺序

第 1 轮：只做“看清”

  1. 收写入口
  2. 收 usecase
  3. 收校验与补全
  4. 收副作用
  5. 收竞态热点

这轮更适合 Pro

第 2 轮：只做“定规则”

  1. 定动作模型
  2. 定副作用归属
  3. 定 quickinput 新角色
  4. 定统一输入内核目录
  5. 定竞态标准

这轮更适合 Think

第 3 轮：只做“试点迁移”

  1. quickinput 接新内核
  2. 选 1 个最常用 view/edit 入口接入
  3. 验证 create / update / delete / validate / normalize / refresh 全链路

这轮更适合 Pro

第 4 轮：才做“全量迁移”

这轮也更适合 Pro

十一、最后给你一句决策建议

你现在该用什么

  • 梳理现状、收集入口、排查副作用、找竞态热点：用 Pro
  • 定规则、定模型、定边界、定验收标准：用 Think

你现在最好的下一步

不是直接改代码，而是先完成这 4 张表：

  1. 写入口清单
  2. usecase 清单
  3. 校验/补全清单
  4. 副作用/竞态清单

这 4 张表一出来，统一输入这件事就从“感觉要改”变成“知道怎么改”。
