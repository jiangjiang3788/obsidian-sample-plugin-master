正文

对，这个我修正一下：

隔天任务不应该靠额外勾选。 你这个例子是对的：

  • 23:00 开始
  • 时长 120 分钟
  • 自动得到 01:00
  • 这本身就应该被识别为隔天任务

所以更合理的规则是：

  • startTime + duration 自动推 endTime，允许自然跨天
  • endTime + duration 倒推 startTime，也允许自然跨天
  • “隔天”不是用户额外维护的状态，而是计算结果
  • 真正需要用户选择的是： 按开始时间向后推，还是 按结束时间向前推

也就是说，保留：

  • timeDirection: forward | backward

不保留：

  • crossDay 勾选框

如果以后真要显示，可以有一个只读派生状态：

  • isCrossDay = true/false

它是算出来的，不是填出来的。

当前阶段实施总计划表（P0 止痛 / P1 稳定 / P2 重构）

说明

本计划表以“先解决真实痛点，再逐步稳定，再考虑重构”为原则。 当前最优先的问题不是 UI 是否优雅，而是：

  • 编辑任务或条目时上下文读取不准
  • 时间计算规则不稳定
  • TaskEntry 与 BlockEntry 的读取模型不一致
  • 时间方向语义不明确

术语约定

  • Entry：统一条目

  • TaskEntry：任务条目

  • BlockEntry：普通块条目 / 信息条目

  • EntryContext：编辑上下文

  • timeDirection：

    • forward：按开始时间向后推
    • backward：按结束时间向前推

关键修正

“隔天任务”不是额外勾选项，而是时间计算的自然结果。 例如：

  • 23:00 + 120 分钟 = 次日 01:00

这本身应自动识别为隔天任务。 因此需要新增的是 timeDirection，而不是 crossDay 勾选框。

当前阶段的真正目标

当前不是追求“架构最优”，而是追求：

  • 编辑不串上下文
  • 时间计算可预测
  • 隔天任务自动正确
  • 任务与块的差异被显式承认
  • 修改范围可控，不陷入无止境重构

一句话执行总纲

先把“编辑的是谁”与“时间到底怎么算”统一，再让字段和结构为这两件事服务，而不是反过来为了重构去拖延止痛。

再把“隔天任务”这件事，给你压成一句实现原则，方便你后面自己审代码：

隔天任务的正确规则

不是：

  • 用户勾选“隔天任务”

而是：

  • 只要时间推导结果跨过 24:00，就自然形成隔天任务

例如：

  • 23:00 + 120 分钟 = 次日 01:00
  • 01:00 - 120 分钟 = 前一日 23:00

所以你真正需要显式控制的是：

  • 按开始时间向后推
  • 按结束时间向前推

不是“是不是隔天”。
