表 2：专门针对“表格编辑 / 拖拽 + 写回”的测试计划（你最在意的那块）

这里我按照你“从顶部到底部”的思路，把E2E → 集成 → 单测三层对齐到同一块功能上。 你只要按顺序实现这些测试，后面怎么重构都不太会白写。

2.1 单测层（最底层：读→改→写协议）

编号 范围 目的 用例描述（可以直接变成 Jest） 关联文件
UT-1 解析（read） 确保按 start/end 区间 + [!thinktxt] 解析稳定 输入：包含 <!-- thinktxt:start --> ... <!-- thinktxt:end --> 以及多条 {key:: value} 行的 markdown；
输出：数组 rows，字段完整（id/title/status/...）。		 parser.ts + DataStore
UT-2 更新（updateRow) 改一个单元格不会破坏其它内容 给定 mdBefore + 操作 updateRow(id=2, patch={title: 'BBB-EDIT'})；
输出 mdAfter：
1）只有目标行内容变；2）缩进/其它行保持不变；3）start/end 注释位置不变。		 patcher（你要写的模块）
UT-3 插入/删除 保证 start/end 块范围仍正确 insertRow(afterId=2, row=...) + deleteRow(id=1)；
验证：新增/删除行都在 start/end 内，区块外内容完全不变。		 同上
UT-4 拖拽排序（reorder） 保证顺序正确、空行/注释不乱 输入 3 行 + 若干空行/注释；操作：reorder(0→2)；
输出：3 行顺序调换，但所有空行/注释保持原位置（或者按你设计的明确规则）。		 同上
UT-5 多块/多区间边界 确保只改目标区间，不污染别的 block 文档里有两个 start/end 区间；对第一个区间执行一组 ops；
验证第二个区间一点没动。		 同上

原则：

  • 单测里只碰字符串，不碰 Obsidian、不碰 UI。
  • 一条用例 = 一个“金样本 md 前后对照”，以后重构 patcher 都靠它。

2.2 集成层（DataStore + patcher + Settings 的闭环）

编号 范围 目的 用例描述 关联模块
IT-1 DataStore + patcher 验证 DataStore 调用 patcher 后能重建正确快照 1. 用虚拟 Vault（内存文件系统）初始化 DataStore。
2. 写入 mdBefore。
3. 调用某个 usecase 或 DataStore API：applyOps(file, ops)。
4. 重新 queryItems；验证顺序/字段与预期一致。		 DataStore + patcher + parser
IT-2 Settings + DataStore 交互 设置改变扫描范围/过滤条件后，读写逻辑仍正确 1. 在 SettingsRepo 里设置过滤条件（比如只看某标签）；
2. 同样执行 applyOps；
3. 再 queryItems；
验证：过滤后结果仍然正确，没掉行或多行。		 SettingsRepository + DataStore
IT-3 多文件场景（跨文件编辑） 将来如有“跨文件仪表盘”，确保写回到正确文件 1. Vault 中有两个文件，各自有一个表格；
2. 对仪表盘中若干行执行 edit + reorder；
3. 验证 DataStore 把改动写回对应文件，而不是乱串。		 同上 +（未来）blocks usecase / layout

2.3 端到端层（你最在意的部分）

这里我假设你会用某种 E2E 框架（WebdriverIO / Playwright / 自写 Obsidian-driver 都可以），重点是“场景”。

场景编号 用户故事 步骤（E2E 自动脚本大纲） 覆盖到的 Phase 事项
E2E-1 打开文档 → 看到 thinktxt 预览表格 1. 启动 Obsidian + 插件。
2. 打开一个包含 start/end + [!thinktxt] 块的示例文档。
3. 切到预览模式。
4. 断言：原来多条 callout 被替换为 1 个 table（行列数正确）。		 Phase2 P1 预览态表格；Phase2 P0 legacy 提取
E2E-2 在仪表盘/侧边视图里编辑一行字段 → 写回原文档 1. 打开仪表盘视图（或你准备用的编辑 UI）。
2. 找到某条记录（title = "AAA"）。
3. 在表格中改成 "AAA-EDIT" 并保存。
4. 打开原 markdown 文档，验证对应行已变更。		 Phase2 P2 表格编辑 + 写回；Phase0 P1 parser 语义
E2E-3 拖拽改变行顺序 → 文本顺序改变 1. 在仪表盘中，抓住第 1 行拖到第 3 行位置。
2. 保存后打开原文档。
3. 验证该表格内的 markdown 行顺序和仪表盘显示一致（对应刚刚的拖拽效果）。		 Phase2 P2 拖拽排序 + 写回；Phase2 P2 patcher
E2E-4 改设置（例如过滤条件） → 仪表盘/预览内容随之变化 1. 在 Settings 里修改 thinktxt 相关设置（比如只显示某 tag / 某路径）。
2. 回到仪表盘或预览。
3. 验证行数/内容发生对应变化。		 Phase2 / Settings；Phase0 P0 测试框架；SettingsRepo
E2E-5 插件 reload/unload → 无遗留订阅/定时器 → 再编辑一轮正常 1. 打开仪表盘，做一次编辑（E2E-2）。
2. 执行插件 unload/reload（或关闭/重开 Obsidian）。
3. 再做一次编辑。
4. 验证：行为正常 + 日志/计数中没有“多次注册”的痕迹。		 Phase0 P2 生命周期收口 + 你未来所有编辑相关功能的安全网

建议顺序：

  1. 先做 E2E-1（纯只读预览）：这个实现成本最低，而且能证明 pipeline 通了。

  2. 然后并行推进：

    • 单测层 UT-1~UT-4（patcher）
    • 集成 IT-1（DataStore+patcher）

  3. patcher 相对稳定后，再补 E2E-2 / E2E-3。

  4. 最后再加 E2E-5（生命周期 + reload），给整套东西兜底。

最后帮你收个口

  • 你这张大表已经把“改什么”写得很清楚了，我现在做的是把“测什么 + 测到哪一层就够”按表结构排好；
