app/ 总体

src/app/
  • 像什么像店长办公室。
  • 一句话它不直接卖东西，它负责安排整家店怎么运转。
  • 它在哪层它在 main 下面、features 上面，属于“调度层”。
  • 主要干什么把 core 里的基础能力接起来，再按规则交给前面的功能区使用。
  • 和 core 的区别app 是安排怎么用，core 是提供能用的东西。
  • 你怎么记看到 app，就想成“店长办公室，不亲自接客，但决定大家怎么干活”。
你后面只要看到“启动、状态、写操作入口、全局装配”这些词，大概率就在 app/ 里。

bootstrap/

src/app/bootstrap/
  • 像什么像开店流程单。
  • 一句话它决定：店一开门，第一步、第二步、第三步分别做什么。
  • 它在哪层属于 app 里面最靠前的部分，因为它负责开场顺序。
  • 主要干什么初始化核心对象、拼装运行时、注册功能、完成启动流程。
  • 和 runtime 的区别bootstrap 管“开店前后的启动顺序”，runtime 管“店开着以后怎么维持现场”。
  • 你怎么记看到 bootstrap，就想成“开店流程单”。
如果你看到 initialize、buildRuntime、register 这类文件名，基本就是在写“先后顺序”。

store/

src/app/store/
  • 像什么像店里的总记录本。
  • 一句话它负责记住：现在店里是什么状态。
  • 它在哪层属于 app 内部的状态中心，是很多功能都会读到的一层。
  • 主要干什么保存全局状态，比如当前设置、界面状态、主题数据、运行中的状态等。
  • 和 usecases 的区别store 负责“存现在是什么样”，usecases 负责“要把它改成什么样”。
  • 你怎么记看到 store，就想成“账本 / 记录本”。
你可以把它理解成“结果放哪”。用户点完东西后，很多变化最后都会落到这里。

usecases/

src/app/usecases/
  • 像什么像店员操作手册。
  • 一句话想做一件事时，不是乱来，而是按这里定义的动作去做。
  • 它在哪层在 features 和 store 中间，是“动作入口层”。
  • 主要干什么把“修改设置、更新主题、触发某个业务动作”这些事情包装成统一入口。
  • 和 features 的区别features 是顾客看到的按钮和界面，usecases 是按钮背后真正执行动作的标准流程。
  • 你怎么记看到 usecases，就想成“操作手册：做事要按流程来”。
这就是为什么项目里强调：features 不要直接改 slices，要走 usecases。因为 usecases 是统一动作入口。

ui/

src/app/ui/
  • 像什么像店长办公室对外的小窗口。
  • 一句话这里不是顾客主要逛的服务区，而是管理层需要露出来的一些界面。
  • 它在哪层属于 app 的外露部分，是 app 少数“要显示出来”的地方。
  • 主要干什么放应用级别的 UI、挂载点、公共模态窗、上下文提供层之类的东西。
  • 和 features 的区别ui 更像系统级界面，features 更像具体功能区界面。
  • 你怎么记看到 ui，就想成“办公室需要露出来的那扇窗”。
如果一个界面是“整个系统都共用的管理外壳”，更可能在 app/ui；如果是某个具体功能自己的界面，更可能在 features 里。

runtime/

src/app/runtime/
  • 像什么像今天营业时的现场安排。
  • 一句话店开起来以后，哪些对象要一直活着、哪些资源要一直维护，就在这里管。
  • 它在哪层属于 app 里面偏“持续运行”的那部分。
  • 主要干什么管理运行期对象、生命周期、可释放资源、当前会话环境之类的内容。
  • 和 bootstrap 的区别bootstrap 负责启动那一瞬间，runtime 负责启动之后的持续状态。
  • 你怎么记看到 runtime，就想成“营业现场一直在运转的安排”。
理解 runtime 最简单的办法就是：它不是“怎么开店”，而是“店开着的时候怎么一直维持”。

capabilities/

src/app/capabilities/
  • 像什么像店里的可用服务清单。
  • 一句话它负责回答：这家店现在到底能提供哪些能力。
  • 它在哪层属于 app 里的“能力组织层”，比具体功能更靠抽象一些。
  • 主要干什么把系统当前可提供的功能、支持的能力、注册关系整理出来，便于上层使用。
  • 和 features 的区别capabilities 更像“这家店能做什么”的清单，features 更像“顾客实际看到的服务区”。
  • 你怎么记看到 capabilities，就想成“服务菜单”。
如果你看到某个地方在整理“系统支持什么”，而不是“顾客点哪个按钮”，那它更像 capabilities。
