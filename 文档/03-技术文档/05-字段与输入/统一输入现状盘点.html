统一输入改造 · 第 1 轮现状盘点报告（只做“看清”）

目标：完成 5 张表的第一版——写入口清单、UseCase 清单、校验/补全清单、副作用清单、竞态热点清单。
范围：基于上传代码包中与下列目录直接相关的实现进行静态盘点：

  • app/ui/components/QuickInputEditor/*
  • features/quickinput/*
  • features/settings/*
  • app/usecases/*
  • app/store/slices/*
  • core/services/InputService.ts
  • core/utils/inputTemplateUtils.ts
  • core/utils/normalize.ts
  • core/types/quickInput.ts
  • 同时补看了与写链直接相连的：core/services/ActionService.ts、core/services/ItemService.ts、core/services/SettingsRepository.ts、core/services/DataStore.ts、platform/modals/*、platform/events/VaultWatcher.ts、app/bootstrap/initializeCore.ts

0. 我认为什么叫“详尽”

这类报告里，“详尽”不是把文件名堆满，而是至少满足这 5 条：

  1. 覆盖够广：你指定的关键目录都看过，而且不是只看文件名，而是能说明“谁在写、写到哪、怎么写”。
  2. 链路够完整：每条关键写入路径都要落到“UI 触发 → UseCase/Service → Store/Repository/Vault → 刷新/通知”。
  3. 证据够硬：重要结论必须能回指到文件和代码位置，而不是凭感觉。
  4. 风险够具体：不仅说“有副作用/有竞态”，还要指出哪条链、为什么会发生、最低处理标准是什么。
  5. 边界够清楚：哪些地方已确认，哪些地方只是高概率推断，要分开写。

按这个标准，这份报告给的是：

  • 覆盖面：已覆盖你指定目录的主干写链，并额外补了直接相连的核心服务/平台事件链。
  • 证据形式：以下所有关键结论都带 文件:行号。
  • 结论粒度：不仅有“清单”，还有“归类”“绕过点”“副作用归属现状”“竞态热点”。

1. 先给结论摘要

1.1 当前系统里，实际上已经存在两条主写链

链 A：设置/配置写链
UI(features/settings 等) → useCases.* → zustand slice / state.updateSettings → SettingsRepository.update() → persistence → repository.subscribe → zustand.setState → UI/运行时订阅副作用

证据：

  • 架构文档要求 features 通过 useCases 写，而不是直接触 slices：app/ARCH_CONSTRAINTS.md:40-59
  • useCases 明确宣称是 UI 唯一业务入口：app/usecases/index.ts:6-22, 76-85
  • SettingsRepository.update() 是统一设置持久化入口：core/services/SettingsRepository.ts:138-159
  • initializeCore() 订阅 SettingsRepository，把 settings 同步回 zustand，并挂接 timer widget / devConsole / categoryColors 等运行时副作用：app/bootstrap/initializeCore.ts:74-130

链 B：记录/输入写链
UI/Modal → InputService / ItemService → Vault 写文件 → DataStore.scanFileByPath()/notifyChange() → 视图刷新

证据：

  • QuickInputModal 提交时直接调 inputService.updateExistingRecord() / inputService.executeTemplate()，并自行 scanFileByPath + notifyChange：platform/modals/QuickInputModal.tsx:492-527
  • InputService 自身直接写 vault；编辑路径里还内置 scanFileByPath + notifyChange：core/services/InputService.ts:16-40, 42-79
  • ItemService.completeItem()/updateItemTime() 也是直接改文件，再 scanFileByPath().then(notifyChange)：core/services/ItemService.ts:33-46, 51-99, 121-149

1.2 你说的风险已经在代码里落地了：“统一写入口”在设置域成立，在输入域不成立

  • 设置域多数动作确实走 useCases.*。
  • 但记录输入域并没有一个 usecase 层的统一动作内核；真实写入核心在 InputService / ItemService，而且入口分散在多个 Modal / View / Timer 流程中。

1.3 文档与现实有两处明显漂移

漂移 1：QuickInputModal 注释与实现不一致
文件头注释写的是“使用 useCases 进行写入操作”，但真实代码直接调用的是 InputService，没有经过任何输入 usecase：

  • 注释：platform/modals/QuickInputModal.tsx:3-6
  • 实现：platform/modals/QuickInputModal.tsx:492-527

漂移 2：Layout/View 规范与实现不一致
架构文档与 layout.usecase.ts 注释都说 features 层应只走 useCases.layout.*，甚至把 viewinstance.usecase 标成“已禁用/不允许 features 直调”：

  • 规范：app/ARCH_CONSTRAINTS.md:142-172, 187-195
  • layout.usecase.ts 注释：app/usecases/layout.usecase.ts:6-18

但现实里：

  • LayoutSettings 直接调用 useCases.viewInstance.createView/updateView：features/settings/LayoutSettings.tsx:105-117, 157-166
  • LayoutEditorPanel 也直接调用 _useCases.viewInstance.createView/updateView：features/settings/components/LayoutEditorPanel.tsx:138, 186
  • ModuleSettingsModal 每次改动都直接 useCases.viewInstance.updateView：features/settings/ModuleSettingsModal.tsx:45-48
  • LayoutRenderer 删除 view 时直接 useCases.viewInstance.deleteView：features/settings/LayoutRenderer.tsx:403-406

更关键的是：layout.usecase.ts 里并没有文档声称的 addView/updateView/deleteView 等 facade 方法；当前 features 只能绕去 viewinstance.usecase.ts。这说明 ViewInstance 侧的“唯一写入口”并没有真正收口。

2. 范围与方法

本轮是静态盘点，不做改码，不做行为回归，不做运行时打点。方法是：

  1. 先读架构约束和 usecase 出口，确定“理论上的唯一写入口”是什么。
  2. 再从 UI 层反查所有写入口：
    • new QuickInputModal(...)
    • useCases.*.*(...)
    • inputService.executeTemplate/updateExistingRecord(...)
    • itemService.completeItem/updateItemTime(...)
  3. 再把这些入口落到实际持久化位置：
    • SettingsRepository.update()
    • vault.writeFile()
  4. 最后补副作用与竞态：
    • scanFileByPath
    • notifyChange
    • createTakeLatest
    • repository/store subscriptions

3. 写入口清单（第一版）

下面的“写入口”分成两大类：

  • A. 记录/输入域：真正影响 md 记录内容
  • B. 设置/配置域：影响模板、主题、布局、视图、AI 设置、计时器设置等

3A. 记录 / 输入域写入口

A-1. QuickInput 命令入口

入口 触发文件 写动作最终去向 备注

平台抽象 openQuickInput() platform/ObsidianModalPort.ts:58-60 同上 平台侧统一打开 modal，但不统一写动作

A-2. 视图/列表中打开 QuickInput 的入口

入口 触发文件 目的 最终写链
任务行点击编辑 shared/ui/items/TaskRow.tsx:34-40 编辑已有任务 QuickInputModal(mode='edit') → InputService.updateExistingRecord()
Block 行点击编辑 shared/ui/items/BlockItem.tsx:26-33 编辑已有 block 记录 同上

Timer 行“编辑任务” features/timer/TimerRow.tsx:56-63 看起来像编辑，但实际不是 edit mode QuickInputModal(create)，只传了 blockId + context，没传 { mode:'edit', editItem }

对 A-2 的判断

TaskRow / BlockItem 是标准编辑链：

  • 直接以 mode:'edit' 打开：shared/ui/items/TaskRow.tsx:37-40、shared/ui/items/BlockItem.tsx:30-33

TimerRow 是一个明显不一致点：

  • 它调用 actionService.getQuickInputConfigForTaskEdit(taskId) 拿到的是 blockId + context：core/services/ActionService.ts:134-168
  • 但真正打开 modal 时并没有传 mode:'edit' 和 editItem：features/timer/TimerRow.tsx:56-63
  • 这意味着它更像“带上下文的新建壳”而不是“回写原记录”。

这个点需要你在第 2 轮定规则时单独裁决：TimerRow 的“编辑任务”究竟应不应该统一到真正的 update 链。

A-3. QuickInputModal 本身就是一个写根，而不是纯壳

QuickInputModal 当前不是一个“只收集输入、交给统一内核处理”的前台壳，而是自己就包含了完整提交逻辑：

  • 读 editorState：platform/modals/QuickInputModal.tsx:492-501
  • 走 onSave 分支（Timer 新任务）：platform/modals/QuickInputModal.tsx:503-506
  • 走 edit 分支：platform/modals/QuickInputModal.tsx:511-516
  • 走 create 分支：platform/modals/QuickInputModal.tsx:517-523
  • toast + close：platform/modals/QuickInputModal.tsx:526-533

所以从改造角度看，QuickInputModal 现在并不是“入口壳”，而是“入口 + 提交控制器 + 副作用协调器”混在一起。

A-4. AI 批量确认也是独立输入写链

AiBatchConfirmModal 不是走 QuickInputModal，而是自己直接：

  • 解析模板：platform/modals/AiBatchConfirmModal.tsx:218-225, 256-260
  • inputService.executeTemplate()：platform/modals/AiBatchConfirmModal.tsx:227-231, 262-264
  • 成功后仅 notifyChange()：platform/modals/AiBatchConfirmModal.tsx:231, 271

这里有两个重要现象：

  1. 它和 QuickInput 共用了模板解析 / form finalize，但没共用提交流程。
  2. 它 create 后没有主动 scanFileByPath()，只 notifyChange()。
    相比之下，QuickInput create 路径会显式 scanFileByPath + notifyChange：platform/modals/QuickInputModal.tsx:518-523

这说明当前“创建记录后的刷新策略”并不统一。

A-5. TimerService 也在直接落库

TimerService.createNewTaskAndStart()：

  • 直接 inputService.executeTemplate(...)：features/timer/TimerService.ts:144-148
  • 手动 dataStore.scanFileByPath(targetFilePath)：features/timer/TimerService.ts:149-153
  • 再决定是 startOrResume(latestItem.id) 还是 notifyChange()：features/timer/TimerService.ts:154-160

这条链本质上也是一个独立创建入口。

A-6. 任务完成/时间修改是另一条记录写链，不走 QuickInput

动作 入口 最终服务
勾选完成任务 features/settings/LayoutRenderer.tsx:376-378 itemService.completeItem()
视图中更新时间 features/settings/LayoutRenderer.tsx:178-180 itemService.updateItemTime()
计时结束写回时长 features/timer/TimerService.ts:124-131 itemService.completeItem() 或 itemService.updateItemTime()

而 ItemService 内部是：

  • 直接读写目标文件行：core/services/ItemService.ts:20-45
  • 完成任务：core/services/ItemService.ts:51-101
  • 更新时间：core/services/ItemService.ts:121-136
  • 写完后 scanFileByPath().then(notifyChange())：core/services/ItemService.ts:45-46, 147-148

A-7. “删除记录”写入口：本轮未找到统一主线

本轮代码盘点里，没有找到与 InputService.updateExistingRecord() 对应的 “deleteExistingRecord” 或统一的记录删除 usecase。

已确认：

  • 配置删除（block/theme/layout/view）很多；
  • 记录编辑/创建很多；
  • 但“删除一条已存在记录”的统一主线，在本轮范围内没有看见。

这不是说系统一定不能删记录，而是说：在当前盘点范围里，没有看到一个成型、显性的、可复用的记录删除主动作。

这对你第 2 轮定动作模型是重要输入：submitDeleteRecord 目前很可能需要新建，而不是简单抽取。

4.2 关于“统一输入”最关键的结论

当前不存在一个对等于 BlocksUseCase / ThemeUseCase / LayoutUseCase 的“InputUseCase”或“RecordInputUseCase”。

也就是说：

  • 配置域已经有 facade；
  • 输入域没有对应 facade；
  • QuickInput / AI Batch / Timer New Task / Item time update 都各自直接连到 service 层。

这就是为什么你现在强调“不要做第二个前台输入渠道，而是先建统一输入内核”是对的：

因为现在缺的不是“再多一个按钮”，缺的是输入域自己的应用层 facade / 动作模型。

  • create 有至少 3 种刷新策略
  • update 有 InputService.updateExistingRecord 与 ItemService.updateItemTime/completeItem 两条子链
  • feedback（Notice / closeModal）散在 UI

已确定 4：ViewInstance 域也需要收口，不然“统一编辑动作”做不稳

否则会出现：

  • 输入域你收口了；
  • 视图编辑域还继续到处 useCases.viewInstance.*；
  • 最后“统一编辑动作”仍然不统一。

11. 我建议你下一轮（定规则）直接回答的 6 个问题

  1. 输入域的新 facade 叫什么？
    例如：recordInput.usecase.ts / inputKernel.usecase.ts / recordCommands.usecase.ts

  2. QuickInputModal 以后是不是只保留“打开、收集、触发提交动作”三件事？

  3. 编辑已有记录时，模板推断逻辑要不要从 modal 抽成独立 resolver？

  4. create 后刷新策略是否统一为：写文件 → scan 指定 path → notifyChange？

  5. ItemService.completeItem/updateItemTime 算不算纳入统一记录动作模型？
    我倾向于算，因为它们本质上也是 record update。

  6. ViewInstance 要不要并回 layout.usecase，还是明确承认它是第二个 facade？
    现在文档和实现不一致，必须选边。

12. 最后一段判断

如果按你的总原则：

不要做第二个前台输入渠道。要做的是：新建统一输入内核，让 quickinput 先接进去。

那基于这轮盘点，我的判断是：

你现在最值得确认的真实现状是

  • 前台入口并不缺，已经太多了；
  • 缺的是输入域统一应用层；
  • QuickInput 现在不是壳，而是半个内核；
  • AI 批量、Timer 新建、Item 时间更新，都是未来必须并入统一动作模型的旁路；
  • 副作用和刷新策略还没收口；
  • 竞态控制只在少数链路做了一半。

所以第 1 轮的结论不是“可以开始大改了”

而是：

现在已经可以开始第 2 轮：定规则。
因为“谁在写、写到哪、哪些地方绕过、哪些副作用在飞、哪些链路有竞态”——已经足够清楚了。
