字段系统深度审计报告与重构方案（架构 / 领域 / 表单 / 上下文 / 筛选全覆盖）

一、报告目标

本报告不是单纯讨论“字段类型有哪些”，而是把字段当成插件的核心骨架进行审视。

字段在当前插件中不是一个 UI 细节，而是贯穿全链路的主轴：

  • 记录数据靠字段组织
  • 表单输入靠字段驱动
  • 展示视图靠字段渲染
  • 筛选和搜索靠字段索引
  • 排序和统计靠字段语义
  • 移动端适配最终也落到字段交互

因此，字段系统如果没有被当成“一级架构对象”来设计，插件后续一定会不断出现：

  • 新字段难加
  • 旧字段行为不一致
  • UI 与数据耦合
  • 移动端和桌面端交互分裂
  • 筛选逻辑越来越散
  • 媒体字段接不进去

二、总判断：字段是否已经到了需要重构的程度

2.1 我的判断

是的，值得重构。

但我不建议“整仓推翻重来”，而建议：

保留已有业务能力与数据资产，重建字段内核、字段运行时和字段渲染协议，再逐步迁移 UI 与表单壳层。

这类重构更像：

  • 不是重写整个插件
  • 而是把“字段”从零散配置、零散判断、零散组件里抽出来
  • 升级为一个统一系统

2.2 为什么字段值得作为重构中心

因为字段本身天然跨越四层：

1. 领域层

字段定义了“记录里有什么”

2. 应用层

字段决定“如何录入、修改、校验、提交、筛选”

3. UI 层

字段决定“怎么显示、怎么编辑、怎么交互”

4. 存储层

字段决定“如何写入 Markdown / frontmatter / 附件 / 派生索引”

如果这四层各自理解字段，且彼此没有统一协议，最终一定会出现一套插件像四套系统拼出来的感觉。

三、什么样的字段审计报告才算“满意的答卷”

一份真正合格的字段审计报告，至少应回答以下问题：

3.1 字段到底是什么

它是：

  • UI 控件配置？
  • 数据 schema？
  • 表单项？
  • frontmatter 键？
  • 筛选维度？
  • 渲染节点？

正确答案是：

字段是一个跨领域、跨应用、跨渲染、跨存储的核心模型。

3.2 字段现在在哪里被使用

必须地毯式盘点：

  • 创建记录
  • 编辑记录
  • 快速输入
  • 详情展示
  • 列表摘要
  • 筛选器
  • 搜索
  • 排序
  • 导出
  • 模板
  • 统计
  • 移动端交互
  • 附件与媒体插入

3.3 字段现在是如何失控的

必须明确回答：

  • 是不是“字段类型”只存在于 UI 层
  • 是不是不同页面对同一字段有不同逻辑
  • 是不是筛选逻辑绕开了字段定义
  • 是不是存储层知道太多 UI 细节
  • 是不是字段行为靠散落的 if/else 驱动

3.4 字段未来如何扩展

必须说明：

  • 新增字段会不会牵一发动全身
  • 图片、音频、附件、多值字段、关联字段能不能接进去
  • 移动端行为能否被字段自身声明

3.5 重构边界在哪里

必须给出：

  • 哪些内容可以保留
  • 哪些必须替换
  • 哪些需要分阶段迁移

如果做不到这几点，报告就只是“看起来很多字”，但不能指导重构。

四、字段在插件中的正确定位

4.1 字段不是表单控件，字段是“记录能力的接口”

字段真正要承载的是：

  • 数据类型
  • 业务语义
  • 校验规则
  • 默认值规则
  • 展示规则
  • 编辑规则
  • 筛选规则
  • 排序规则
  • 存储规则
  • 平台行为

所以字段的定义不能只是：

{
  key: 'title',
  type: 'text',
  label: '标题'
}

这太薄了。它只能支撑一个输入框，支撑不了一个系统。

4.2 字段应该被定义成“能力对象”

更合理的方向是：

type FieldDefinition = {
  key: string;
  kind: FieldKind;
  label: string;
  description?: string;

  cardinality: 'single' | 'multiple';
  required?: boolean;

  defaultValue?: DefaultValuePolicy;
  validate?: Validator[];
  normalize?: Normalizer[];

  storage: FieldStoragePolicy;
  display: FieldDisplayPolicy;
  input: FieldInputPolicy;
  filter: FieldFilterPolicy;
  sort?: FieldSortPolicy;

  mobile?: MobileFieldPolicy;
  desktop?: DesktopFieldPolicy;
};

字段从“配置项”升级成“协议对象”。

五、字段的地毯式使用场景审计

5.1 输入场景

字段在输入里至少影响：

  • 使用什么控件
  • 单行还是多行
  • 是否自动增高
  • 回车语义是什么
  • 是否需要建议值
  • 是否需要 sheet 选择
  • 是否允许为空
  • 是否即时校验
  • 是否要保证光标可见
  • 是否支持移动端专属行为

核心判断

如果这些逻辑今天都散在组件里，而不是挂在字段定义上，那么字段系统还没有成型。

5.2 表单上下文

字段不仅决定单个输入框，还决定表单上下文：

  • 字段顺序
  • 字段分组
  • 哪些字段默认折叠
  • 哪些字段是高级项
  • 哪些字段属于危险区
  • 哪些字段在某个记录类型下隐藏
  • 哪些字段互相联动

关键问题

现在你的代码里，如果“字段之间关系”不是被 schema 或规则系统表达，而是页面里手写判断，那就说明表单上下文设计偏弱。

5.3 展示上下文

同一个字段在不同视图里，展示方式不一样：

  • 列表视图：只显示摘要
  • 详情视图：显示完整值
  • 卡片视图：显示重点信息
  • 时间轴视图：可能只显示时间与标签
  • 移动端：可能需要更紧凑的显示规则

所以字段必须能描述：

  • 摘要显示
  • 完整显示
  • 空值显示
  • 截断规则
  • 富显示规则
  • 媒体预览规则

5.4 筛选上下文

筛选不应该是“页面自己理解字段”。

字段自身必须声明：

  • 是否可筛选
  • 用什么操作符
  • 单值还是多值
  • 文本是 contains 还是 equals
  • 日期是区间还是单点
  • 标签是任意匹配还是全匹配
  • 媒体字段是否可筛“是否存在”

这是重构重点

如果你不把筛选并回字段系统，后续筛选器一定越来越像另一套私有逻辑。

5.5 排序与聚合上下文

很多插件一开始只想到“字段输入”，后面才想“字段排序”“字段统计”，这时会发现字段定义不够。

字段应预留：

  • 是否可排序
  • 默认排序算法
  • 大小写规则
  • 日期排序
  • 数值排序
  • 多值字段排序策略
  • 是否可聚合统计
  • 聚合方式有哪些

5.6 存储上下文

字段最终要落到 Obsidian 的文件系统与 Markdown 上。

官方建议插件优先用 Vault API，而不是直接用 Adapter；修改 frontmatter 应用 FileManager.processFrontMatter，而不是手动读写 YAML。移动端也不能假定桌面文件系统行为，因为移动端 Vault.adapter 可能是 CapacitorAdapter，不是桌面文件系统适配器。(Developer Documentation)

因此字段系统必须明确定义存储策略：

  • frontmatter
  • inline markdown
  • note body block
  • linked attachment
  • derived index
  • plugin data cache

六、字段重构的核心判断：到底要不要推翻

6.1 不建议推翻的部分

以下内容一般可以保留：

  • 已有业务语义
  • 已有字段 key
  • 已有记录数据
  • 已有用户工作流
  • 已有前台页面入口
  • 已有 Markdown 存储大方向

6.2 建议重做的部分

以下部分值得重做：

  • 字段定义协议
  • 字段渲染注册机制
  • 字段输入行为层
  • 字段筛选协议
  • 字段展示协议
  • 字段存储映射层
  • 移动端字段行为模型

6.3 结论

所以这不是“推翻整个插件”，而是：

推翻旧的字段实现方式，保留旧的业务目标和数据世界。

这类重构是值得的，而且性价比高。

七、建议的新字段架构

7.1 四层结构

第一层：Field Definition

描述字段“是什么”

第二层：Field Runtime

描述字段“此刻如何运行”

第三层：Field Renderer

描述字段“如何展示/编辑”

第四层：Field Persistence

描述字段“如何落盘与读取”

7.2 建议的数据模型

type FieldKind =
  | 'text'
  | 'textarea'
  | 'number'
  | 'select'
  | 'multiselect'
  | 'date'
  | 'datetime'
  | 'toggle'
  | 'checkbox'
  | 'radio'
  | 'tags'
  | 'richtext'
  | 'image'
  | 'audio'
  | 'attachments';

字段定义层

type FieldDefinition = {
  key: string;
  kind: FieldKind;
  label: string;
  required?: boolean;
  cardinality?: 'single' | 'multiple';

  defaultValue?: unknown;
  validate?: FieldValidator[];
  normalize?: FieldNormalizer[];

  storage: FieldStoragePolicy;
  display: FieldDisplayPolicy;
  input: FieldInputPolicy;
  filter?: FieldFilterPolicy;
  mobile?: MobileFieldPolicy;
};

运行时层

type FieldRuntimeState = {
  visible: boolean;
  disabled: boolean;
  readonly: boolean;
  dirty: boolean;
  touched: boolean;
  errors: string[];
};

渲染协议层

type FieldRendererRegistry = {
  display: Record<FieldKind, DisplayRenderer>;
  editor: Record<FieldKind, EditorRenderer>;
  filter: Record<FieldKind, FilterRenderer>;
};

8.2 分层职责

domain/fields

不碰 DOM，不碰平台 API，只定义字段语义

application/fields

把字段应用到表单、筛选、序列化、反序列化

presentation/fields

承接字段运行时状态，如错误、焦点、可见性、移动端输入状态

ui/fields

具体渲染为按钮、输入框、标签、sheet、预览卡片

九、字段在输入与表单中的重构建议

9.1 统一“字段输入协议”

每个字段都应声明：

type FieldInputPolicy = {
  editor: 'inline' | 'sheet' | 'fullscreen';
  enterBehavior?: 'newline' | 'next' | 'submit';
  autoGrow?: boolean;
  keepCursorVisible?: boolean;
  placeholder?: string;
  suggestions?: string[] | SuggestionSource;
};

9.2 字段不直接决定组件，字段决定“输入策略”

例如：

  • text 不等于必须用 <input>
  • select 不等于必须下拉框
  • image 不等于必须上传按钮

真正应该决定的是：

  • 编辑场景
  • 平台策略
  • 是否需要全屏编辑态
  • 是否需要预览

十、字段在上下文中的重构建议

10.1 字段上下文不是平铺数组

建议引入：

type FieldGroup = {
  id: string;
  title: string;
  fields: string[];
  collapsible?: boolean;
  priority?: 'primary' | 'secondary' | 'advanced' | 'danger';
};

这样字段在表单中不再只是平铺，而是进入上下文。

10.2 条件可见与联动

建议用规则而不是组件里写死：

type VisibilityRule = (record: DraftRecord) => boolean;
type EnableRule = (record: DraftRecord) => boolean;

十一、字段在筛选中的重构建议

11.1 字段必须自己声明筛选语义

例如：

type FieldFilterPolicy = {
  enabled: boolean;
  operators: string[];
  quickFilter?: boolean;
};

11.2 各字段建议的筛选方式

  • text：contains / equals / startsWith
  • textarea：contains
  • number：eq / gt / lt / between
  • date：before / after / between
  • select：equals / in
  • multiselect：containsAny / containsAll
  • tags：containsAny / containsAll
  • image：exists / notExists
  • audio：exists / notExists / durationRange（以后扩展）

十二、图片与音频字段的设计占位

这一部分你说“不着急”，那我就按“先预留正确接口，不急着做全功能”来设计。

12.1 为什么图片 / 音频字段值得提前建模

Obsidian 官方帮助文档明确支持把附件放进 vault，附件可以是图片、音频、PDF 等；也支持把这些文件嵌入到笔记中。帮助文档还列出了支持的图片与音频格式，例如图片支持 .avif .bmp .gif .jpeg .jpg .png .svg .webp，音频支持 .flac .m4a .mp3 .ogg .wav .webm .3gp。不过官方也说明，音频、视频等文件的实际支持还取决于设备可用的 codec。(Obsidian)

12.2 图片 / 音频字段不应直接存二进制内容

建议不要把图片、音频本体塞进字段 JSON 里，而是：

  • 文件本体存 vault 附件
  • 字段存“引用”

因为 Obsidian 本身就以 vault 中的附件文件和内部嵌入为基础。官方帮助文档说明，附件是 vault 里的常规文件，桌面端可以拖放嵌入，移动端用附件选项插入或链接。(Obsidian)

12.3 推荐的字段值结构

图片字段

type ImageFieldValue = {
  path: string;
  alt?: string;
  caption?: string;
  width?: number;
  height?: number;
};

音频字段

type AudioFieldValue = {
  path: string;
  title?: string;
  durationMs?: number;
  transcriptPath?: string;
};

多媒体集合

type MediaListValue<T> = {
  items: T[];
};

12.4 存储策略建议

有两条路线：

路线 A：字段值写 frontmatter，只存 path

优点：

  • 结构化
  • 便于筛选和程序读取

缺点：

  • 真正显示时仍要转成 embed 或预览

路线 B：字段值写 frontmatter + 正文渲染 embed

例如图片可转为内部嵌入引用，音频同理。官方帮助文档说明内部嵌入语法是 ![[filename]] 这一类的嵌入方式。(Obsidian)

我的建议是：

先走路线 A，保守、稳定、可筛选。 以后再在展示层补 route B 的富渲染。

12.5 文件创建与移动的设计建议

官方对插件 API 的指导是优先用 Vault API，不要优先操作 Adapter；修改 frontmatter 用 processFrontMatter。另外，附件的默认位置由 Obsidian 的“Default location for new attachments”设置决定。(Obsidian)

所以图片 / 音频字段的设计建议是：

  • 上传或接收文件后，统一通过 vault 流程落盘
  • 字段层只保留路径引用
  • frontmatter 的修改通过 processFrontMatter
  • 不让 UI 组件自己决定文件放哪
  • 增加一个 AttachmentPlacementService

十三、字段系统最应该避免的坑

1. 字段定义只存在于 UI 层

结果是业务、筛选、存储都会分叉。

2. 字段 key 与存储 key 混乱

后面迁移极痛苦。

3. 字段类型太少，行为全靠 if/else

短期省事，长期爆炸。

4. 让每个页面自己理解字段

最终每个页面都有一套“半私有字段逻辑”。

5. 媒体字段直接绑死某个平台实现

移动端和桌面端就会裂开。

十四、推荐的重构策略

第一阶段：字段盘点

输出一张矩阵：

  • 字段 key
  • 当前类型
  • 当前存储方式
  • 当前展示位置
  • 当前输入方式
  • 当前筛选方式
  • 是否移动端特殊
  • 是否存在重复逻辑

第二阶段：建立新字段协议

完成：

  • FieldDefinition
  • FieldStoragePolicy
  • FieldInputPolicy
  • FieldDisplayPolicy
  • FieldFilterPolicy

第三阶段：建立渲染注册表

  • 显示渲染器
  • 编辑渲染器
  • 筛选渲染器

第四阶段：替换旧页面

按场景替换：

  1. QuickInput
  2. Full Editor
  3. Detail View
  4. Filter Panel
  5. List Summary

第五阶段：扩展媒体字段

先支持：

  • 图片 path 引用
  • 音频 path 引用
  • 基础预览
  • exists 筛选

不急着做：

  • 转码
  • 波形
  • 音频剪辑
  • OCR
  • 自动转写

十五、最终结论

你现在要做的，不是“再给字段加两个 type”。

你真正要做的是：

把字段从“零散表单项”升级成“插件核心协议层”。

这是一次很值得做的重构，而且不必推翻整个插件。

最合理的方向是：

  • 保留业务与数据
  • 重做字段内核
  • 重组字段运行时
  • 重建字段渲染与筛选协议
  • 为图片、音频等附件字段预留路径引用型设计

十六、一句话结论

字段不是表单里的一个输入项，而是贯穿记录、表单、展示、筛选、存储和平台适配的核心对象；你这次重构最应该重建的，不是页面，而是字段系统本身。
