开发指南

本文面向“要在本仓库继续开发/迭代功能的人”，目标是：新同学能在 30 分钟内跑起来、能定位代码入口、能通过质量门禁（gate）。

1. 环境准备

  • Node.js + npm
  • Obsidian 桌面端（用于加载本地插件目录）

安装依赖：

npm install

2. 常用命令

# 质量门禁（lint + typecheck + 各类架构/边界 gate）
npm run check

# 测试
npm test

# 构建（产物到 dist/，并复制到仓库根目录 main.js/styles.css 以便发布/拷贝）
npm run build

如果 npm run check 失败，通常是触发了 scripts/*gate*.mjs 的边界约束；先看报错提示对应哪条 gate。

3. 本地加载到 Obsidian（开发/验证）

Obsidian 插件目录结构（最小）：

  • manifest.json
  • main.js
  • styles.css

把它们拷贝到：

<你的Vault>/.obsidian/plugins/think-os/

PowerShell 示例：

$vault = "D:\ObsidianVault"
$pluginDir = Join-Path $vault ".obsidian\plugins\think-os"
New-Item -ItemType Directory -Force -Path $pluginDir | Out-Null
Copy-Item -Force manifest.json, main.js, styles.css $pluginDir

4. 代码入口与运行时

  • 插件入口：src/main.ts
    • onload()：加载设置、配置 DI 容器、启动 ServiceManager、注册命令、注册 thinktxt 预览渲染等
  • 平台适配层：src/platform/
    • 作为 Obsidian API 的主要入口（Vault/UI/Modal/Events/Metadata/FileStat/MessageRender 等端口实现）

5. 项目结构（建议理解顺序）

  • src/app/：应用装配与运行时（ServiceManager、usecases、capabilities、FeatureLoader 等）
  • src/core/：核心领域能力（settings/storage/ports 等）
  • src/features/：具体功能（timer / quickinput / aiinput / aichat / settings ...）
  • src/shared/：跨模块共享工具与公共 UI（含 public/private 约束）

6. 架构边界与约束（重要）

仓库中存在多条 gate 脚本用于约束依赖方向、public API、平台泄漏等（见 package.json 的 gate 脚本以及 scripts/）。

开发时的经验法则：

  • 功能优先落在 src/features/<name>/，对外暴露最小接口，减少跨 feature 直接依赖。
  • 避免在业务代码里直接调用 Obsidian API：优先通过 src/platform/ 的 ports 访问（便于测试与演进）。
  • 尊重 public/private 边界：如果导入路径导致 gate 报错，通常意味着“你跨层了/跨模块了/依赖方向不对”。

7. 新增功能的一般步骤（模板）

  1. 在 src/features/<featureName>/ 新建模块（服务/用例/UI）
  2. 在 DI 容器中注册（参考现有 feature 的注册方式）
  3. 在 src/main.ts 或 FeatureLoader 侧接入入口（命令、视图、setting tab 等）
  4. 补齐最小测试（至少覆盖核心纯逻辑）
  5. 跑 npm run check + npm test + npm run build

8. 提交与协作建议

  • 小步提交，信息清晰（例如：feat(timer): ... / fix(aiinput): ...）
  • 合并前给出：
    • 为什么改
    • 怎么测的（命令/操作路径）
    • 如果涉及 UI，附截图/录屏
