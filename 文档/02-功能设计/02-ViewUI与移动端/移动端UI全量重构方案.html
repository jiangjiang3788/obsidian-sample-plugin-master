移动端 UI 全量重构方案（地毯式覆盖版）

一、目标定义

本次重构的目标不是“把桌面端界面缩到手机里”，而是为移动端建立一套独立、完整、可维护的 UI 体系。

目标

  • 让移动端成为独立可用的一等体验
  • 建立桌面端与移动端分治的 UI 架构
  • 统一交互规则、组件规范、状态模型和适配策略
  • 解决当前由补丁式兼容带来的反复问题
  • 为后续功能扩展建立稳定基础

非目标

  • 不追求所有桌面端交互在移动端一比一复刻
  • 不追求所有页面同构
  • 不优先做视觉翻新，优先做结构稳定和交互正确

二、核心判断

2.1 当前问题的本质

当前移动端问题不是单点 bug，而是系统性缺陷：

  • 桌面端优先设计，移动端后补
  • 页面、弹窗、表单、选择器、操作区没有移动优先建模
  • 交互逻辑散落在组件内部
  • 布局、滚动、键盘、安全区、焦点、状态耦合过深
  • 缺乏统一的移动端设计语言和适配层

2.2 重构的原则

移动端 UI 重构必须遵循以下原则：

  1. 移动端独立建模
  2. 共享业务，不共享壳层
  3. 单一主滚动容器
  4. 输入、选择、提交是三类核心场景
  5. 状态先行，样式其次
  6. 视觉统一服从交互正确
  7. 可调试、可验证、可回归

三、产品层设计：移动端不应只是桌面缩略版

3.1 双端产品形态定义

桌面端适合：

  • 高信息密度
  • 多区块并排
  • hover、快捷键、右键
  • 长时间编辑与切换
  • modal、side panel、dense table

移动端适合：

  • 单任务流
  • 明确上下文
  • 一次一个重点
  • 焦点驱动交互
  • 全屏页、sheet、底部动作区、分步式输入

结论

同一个业务能力，桌面端与移动端应共享：

  • 领域模型
  • 表单定义
  • 提交逻辑
  • 校验规则

但不应强行共享：

  • 页面骨架
  • 布局形态
  • 滚动策略
  • 操作位置
  • 输入与选择交互

四、整体信息架构重构

4.1 页面层级建议

移动端建议将 UI 拆成四种一级形态：

A. 全屏页面 Full Screen View

适用于：

  • 主功能页
  • 长表单
  • 设置页
  • 详情页
  • 编辑页

特点：

  • 独立导航栏
  • 单一主滚动区
  • 支持返回
  • 适合复杂内容

B. 底部抽屉 Bottom Sheet

适用于：

  • 轻量选择
  • 次级操作
  • 筛选
  • 短表单
  • 二次确认

特点：

  • 轻量
  • 临时
  • 强上下文依附
  • 不适合重编辑

C. 全屏编辑态 Full Screen Editor

适用于：

  • 富文本输入
  • 长文本输入
  • 多字段编辑
  • 需要键盘长期占用的场景

特点：

  • 输入优先
  • 操作按钮简化
  • 减少顶部/底部占位
  • 保证光标与内容安全可见

D. 操作菜单 Action Surface

适用于：

  • 单选/多选
  • 排序
  • 批量操作
  • 分享/删除/归档等动作

特点：

  • 不承载复杂内容
  • 优先使用动作列表
  • 避免在移动端塞复杂弹窗

4.2 信息架构分区

每个移动端页面建议统一分为：

  1. 顶部导航区
  2. 状态/上下文区
  3. 主内容区
  4. 底部动作区
  5. 系统反馈区

其中最重要的约束：

  • 主内容区必须是唯一主滚动区
  • 顶部和底部是否固定，要按页面类型明确，不可混搭

五、页面骨架系统

5.1 页面骨架基类

建议抽象以下壳层组件：

MobileScreenShell
MobileSheetShell
MobileEditorShell
MobileSelectionShell

每种壳层各自负责：

  • safe area
  • header
  • body
  • footer
  • scroll region
  • keyboard behavior
  • dismiss behavior

这样业务组件不再自己处理这些问题。

5.2 Header 体系

移动端顶部区域需要统一规范：

Header 类型

  • Standard Header：标题 + 返回
  • Context Header：标题 + 副标题/状态
  • Editor Header：返回/关闭 + 保存
  • Search Header：搜索框 + 取消

规则

  • 标题不宜超过两行
  • 主操作不超过 1 个
  • 次操作通过 menu 收纳
  • 不在 header 堆按钮

5.3 Footer / Bottom Action Bar 体系

Footer 类型

  • 固定底部主操作
  • 吸附型次操作区
  • 键盘共存型底部操作区
  • 无 footer 页面

规则

  • 输入页避免厚重固定 footer
  • 只有主动作才长期固定
  • 次动作收进更多菜单
  • 键盘出现时 footer 要么弱化，要么让位，不与输入区争空间

六、导航系统重构

6.1 移动端导航原则

移动端不应复制桌面端多面板结构，而应优先：

  • 栈式导航
  • 返回优先
  • 当前任务优先
  • 上下文可回溯

推荐结构

  • 一级：主入口
  • 二级：列表/模块页
  • 三级：详情/编辑页
  • 临时操作：sheet

6.2 面包屑与路径

桌面端常见面包屑在移动端不宜保留为完整路径串。 建议改为：

  • 返回按钮
  • 简短标题
  • 必要时副标题展示父级信息

七、列表系统重构

7.1 移动端列表不应照搬桌面表格

桌面端擅长：

  • 表格
  • 多列
  • 高密度信息

移动端应改成：

  • 卡片列表
  • 两层信息结构
  • 重要信息前置
  • 次信息折叠

7.2 列表项设计规范

每个列表项建议限制为：

  • 主标题
  • 一行摘要
  • 一行元信息
  • 0~2 个轻操作

不要做：

  • 每行 4 个按钮
  • 多列复杂对齐
  • 小字号堆信息

可做：

  • 左主信息，右状态
  • 长按进入批量操作
  • 左滑/右滑触发轻操作，但必须有显式替代入口

7.3 列表状态

每个列表页都必须具备完整状态体系：

  • 加载中
  • 空状态
  • 错误状态
  • 无权限
  • 筛选结果为空
  • 局部刷新
  • 加载更多

八、表单系统重构

8.1 表单是移动端重灾区，必须单独设计

表单拆为三层：

  1. 字段层
  2. 分组层
  3. 页面层

8.2 字段类型体系

建议统一字段模型：

type FieldKind =
  | 'text'
  | 'textarea'
  | 'number'
  | 'select'
  | 'multiselect'
  | 'date'
  | 'datetime'
  | 'toggle'
  | 'checkbox'
  | 'radio'
  | 'tags'
  | 'richtext'
  | 'attachments'

8.3 字段行为模型

每个字段应可声明移动端行为：

type MobileFieldBehavior = {
  autoGrow?: boolean;
  openInSheet?: boolean;
  enterBehavior?: 'newline' | 'next' | 'submit';
  keepCursorVisible?: boolean;
  selectionStyle?: 'inline' | 'sheet' | 'fullscreen';
};

8.4 表单布局规则

桌面端常见：

  • 两列或多列
  • label 左对齐或上对齐均可
  • 可做紧凑布局

移动端建议：

  • 单列
  • label 在上
  • 字段之间有明确呼吸感
  • 分组之间有明显层次
  • 不做并排双列字段，除非极简场景

8.5 分组策略

长表单必须分组：

  • 基本信息
  • 详细信息
  • 高级选项
  • 危险操作

并支持：

  • 折叠/展开
  • 锚点跳转
  • 分段保存

8.6 验证与错误提示

移动端表单错误提示必须满足：

  • 错误信息贴近字段
  • 不只靠颜色
  • 页面顶部有汇总提示时，也要能跳到字段
  • 提交失败时自动滚动到第一个错误字段

九、输入体验系统重构

9.1 输入保护能力独立化

建议抽象为：

useInputVisibilityGuard()
useKeyboardState()
useAutoGrowTextarea()
useFieldFocusNavigator()

负责内容

  • focus 管理
  • keyboard suspected/detected
  • 光标可见性
  • 内容安全区
  • 多次滚动校正
  • textarea 自动增高
  • 调试输出

9.2 输入目标等级

输入体验至少分三层：

  1. 控件可见
  2. 光标可见
  3. 当前输入内容可见

系统设计目标必须至少达到第 3 层。

9.3 回车语义

移动端必须按字段类型定义：

  • 单行 text：next / done
  • textarea：newline
  • 搜索框：search
  • 表单最后字段：done 或 submit

不要让所有字段共用同一种回车语义。

十、选择器与弹层系统重构

10.1 选择器分类

轻量选择

  • 单选项少
  • 直接 sheet

中量选择

  • 需要搜索
  • 全屏选择页

重量选择

  • 多条件筛选
  • 全屏筛选器，不放在小 modal 里

10.2 弹层体系分级

建议统一定义四类弹层：

  1. Toast / Notice
  2. Popover / Action Menu
  3. Bottom Sheet
  4. Full Screen Modal

规则

  • 移动端少用居中 modal
  • 复杂交互尽量升级到全屏页或全屏编辑态
  • 不在移动端堆嵌套弹窗

十一、详情页系统重构

11.1 详情页结构

详情页建议统一为：

  • 顶部主信息
  • 关键状态
  • 分组内容
  • 关联操作
  • 历史/日志
  • 底部主动作

规则

  • 关键信息前置
  • 长内容分块
  • 二级操作通过 menu / sheet 收纳
  • 危险操作下沉到底部独立区块

十二、搜索与筛选系统重构

12.1 搜索页

移动端搜索应采用独立 search mode：

  • 顶部搜索框
  • 实时结果
  • 最近搜索
  • 清空与取消
  • 支持空查询态

12.2 筛选器

复杂筛选建议：

  • 不嵌在小弹窗
  • 使用底部 sheet 或全屏筛选页
  • 支持“重置 / 应用”
  • 显示当前已生效筛选标签

十三、反馈系统重构

13.1 反馈层级

轻反馈

  • toast
  • inline hint

中反馈

  • section error
  • banner

重反馈

  • full-screen error
  • blocking confirm

原则

反馈强度必须与问题严重程度一致。

13.2 Notice 使用规则

避免无意义提示，例如：

  • “核心功能已加载”
  • “组件初始化成功”

用户只应看到与任务有关的反馈：

  • 已保存
  • 保存失败
  • 网络不可用
  • 已复制
  • 已删除

十四、空态与错误态设计

每个页面都应具备以下 UI 状态设计：

  • 首次加载
  • 部分加载
  • 空数据
  • 搜索为空
  • 筛选为空
  • 异常错误
  • 离线
  • 权限不足

空态应包含

  • 状态说明
  • 一句指导
  • 一个主动作

十五、设计系统与视觉系统

15.1 Token 体系

建立统一设计 token：

  • spacing
  • radius
  • font size
  • line height
  • color
  • shadow
  • z-index
  • safe-area inset
  • animation duration

15.2 间距规则

移动端建议建立一套固定尺度，例如：

  • 4 / 8 / 12 / 16 / 20 / 24 / 32

不要靠页面临时拍脑袋。

15.3 字号层级

建议控制在有限层级内：

  • title
  • subtitle
  • body
  • caption
  • meta

移动端不宜字号过多。

15.4 触控热区

移动端所有可点击元素必须满足足够热区。 优先保证可点，不优先追求视觉紧凑。

十六、组件系统重构

16.1 组件分层

基础组件

  • Button
  • Input
  • Textarea
  • Select
  • Toggle
  • Chip
  • Card
  • Sheet
  • Banner

业务组合组件

  • FieldRow
  • SectionHeader
  • ActionBar
  • StatusPill
  • SearchHeader
  • ListItemCard

场景组件

  • QuickInputMobileView
  • MobileFilterPanel
  • MobileEditorPage

16.2 组件约束

每个组件必须说明：

  • 何时使用
  • 何时不要用
  • 桌面端与移动端差异
  • 是否支持 keyboard-safe
  • 是否支持 safe-area
  • 是否支持单一滚动容器环境

十七、架构设计：共享业务，不共享壳层

17.1 推荐目录结构

src/
  domain/
    quickInput/
    records/
    search/
  application/
    quickInput/
    navigation/
  presentation/
    viewmodels/
    mobile/
    desktop/
  ui/
    shared/
    mobile/
    desktop/
    shells/
    components/
  infra/
    platform/
    storage/
    debug/

17.2 分层职责

domain

  • 业务对象
  • 校验
  • 领域规则

application

  • 用例
  • 流程
  • 命令/提交协调

presentation

  • 页面状态
  • 选择器状态
  • keyboard / focus / scroll 状态

ui

  • 组件与布局

infra

  • 平台适配
  • 宿主能力
  • 调试日志

十八、移动适配层设计

18.1 必须有独立 platform adapter

建议抽象：

platform/mobile/
  viewport.ts
  keyboard.ts
  safeArea.ts
  gestures.ts
  haptics.ts
  clipboard.ts

目的

把平台差异隔离，不让业务组件直接接触宿主不稳定 API。

18.2 keyboard 适配原则

  • 不能只依赖 visualViewport
  • 需要 suspected + detected 双模型
  • 需要 focus 驱动兜底
  • 需要调试出口

十九、状态管理设计

19.1 状态分类

业务状态

  • 表单值
  • 提交结果
  • 数据内容

UI 状态

  • 当前激活字段
  • sheet 是否打开
  • keyboard 是否 suspected
  • 当前滚动目标
  • 是否处于 selection mode

派生状态

  • 是否可提交
  • 是否有未保存修改
  • 是否存在错误字段
  • 当前页面主动作是否可用

不要把这三类混在一起。

二十、性能与渲染治理

20.1 移动端性能重点

  • 避免长列表一次性渲染
  • 减少 layout thrash
  • 避免输入时大量状态联动
  • 滚动中不做重计算
  • 减少多层阴影和复杂动画

20.2 输入场景性能规则

  • textarea auto-grow 节流
  • focus 保护逻辑最少化 DOM 查询
  • 调试模式与正式模式分离
  • 键盘期间避免全页重渲染

二十一、调试与观测体系

21.1 移动端必须有无控制台调试能力

提供：

  • 调试面板
  • 一键复制日志
  • 导出日志文件
  • 当前视口/焦点/滚动快照

21.2 每个复杂 UI 能力都要有可观测数据

例如：

  • 当前 page shell 类型
  • 当前滚动容器
  • 当前 keyboard 状态
  • 当前 safe area
  • 当前 active field
  • 最近交互事件

二十二、测试体系

22.1 测试层次

单元测试

  • 领域规则
  • 表单校验
  • 状态派生

组件测试

  • 字段渲染
  • 错误提示
  • auto-grow
  • bottom sheet 行为

交互测试

  • focus 与滚动
  • 提交失败跳错
  • 列表筛选
  • 搜索与取消

设备测试

  • iOS
  • Android
  • 小屏/大屏
  • 横屏/竖屏

22.2 回归清单

每次移动端改动至少检查：

  • 页面是否只有一个主滚动区
  • 键盘出现时内容是否安全可见
  • footer 是否与输入区冲突
  • sheet 是否可关闭/回退
  • 长列表是否流畅
  • 表单错误是否能定位
  • safe area 是否正确

二十三、治理规范

23.1 禁止项

  • 禁止把移动端兼容写成零散 CSS 补丁
  • 禁止一个页面多个主滚动容器
  • 禁止复杂交互放进小居中 modal
  • 禁止业务组件直接读不稳定平台 API
  • 禁止通知泛滥

23.2 必做项

  • 新页面必须声明 shell 类型
  • 新字段必须声明移动行为
  • 新弹层必须声明层级
  • 新表单必须有错误态设计
  • 新交互必须有回退路径

二十四、落地路线图

阶段 1：止血与抽象

  • 修复输入、候选栏、auto-grow
  • 建立 keyboard/focus/safe-area 适配层
  • 建立单一滚动容器约束

阶段 2：壳层重构

  • 落地 MobileScreenShell / MobileSheetShell / MobileEditorShell
  • 将现有移动端页面逐步迁移

阶段 3：组件体系重构

  • 统一字段、列表项、header、footer、sheet
  • 建立移动端组件库与规范

阶段 4：双端分治

  • 桌面与移动分离为两套展示骨架
  • 保持业务层共享

阶段 5：规范治理

  • 建立 review checklist
  • 建立调试面板
  • 建立回归测试矩阵

二十五、最终结论

完整移动端重构，不应理解为“修几个样式”或“兼容键盘”。

它本质上是一次从产品形态到架构边界的重建：

  • 从桌面优先转向双端并重
  • 从补丁兼容转向系统建模
  • 从页面堆砌转向壳层治理
  • 从视觉修补转向交互正确
  • 从单点修复转向全链路可维护

二十六、一句话总纲

移动端 UI 的正确重构方式，不是把桌面界面塞进手机，而是围绕“单任务、单滚动、输入优先、选择清晰、反馈克制、状态可观测”重新建立一套完整系统。
