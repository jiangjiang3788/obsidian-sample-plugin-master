Think OS 功能清单

本文档记录项目当前已实现的功能，作为需求管理的基线参考。 最后更新：2026-02-28

1. 项目基本信息

  • 插件名称：Think OS
  • 插件 ID：think-os
  • 技术栈：TypeScript + Preact + Vite + Jest
  • 定位：基于 Preact 的 Obsidian 仪表盘/效率工具插件
  • 入口文件：src/main.ts

2. 已实现的核心功能

2.1 Dashboard（仪表盘）

  • 功能描述：以视图形式组织插件 UI
  • 技术实现：使用 Preact 构建交互界面
  • 位置：src/app/ + src/features/
  • 状态：✅ 已实现

2.2 Timer（计时器）

  • 功能描述：计时器管理功能
  • 核心能力：
    • 启动/停止计时
    • 计时数据写回
    • 命令行操作支持
  • 位置：src/features/timer/
  • 状态：✅ 已实现

2.3 Quick Input（快速输入）

  • 功能描述：快速输入相关能力
  • 位置：src/features/quickinput/
  • 状态：✅ 已实现

2.4 AI Input（AI 输入）

  • 功能描述：AI 辅助输入功能
  • 位置：src/features/aiinput/
  • 状态：✅ 已实现

2.5 AI Chat（AI 对话）

  • 功能描述：AI 对话能力
  • 位置：src/features/aichat/
  • 状态：✅ 已实现

2.6 Thinktxt 预览渲染

  • 功能描述：将连续的 [!thinktxt] callout 在预览模式下合并渲染为只读表格
  • 核心能力：
    • 识别 [!thinktxt] callout 块
    • 预览模式下渲染为表格
    • 只读展示
  • 位置：src/core/utils/parser.ts + platform 注册
  • 状态：🟡 部分实现（预览渲染待完善）

2.7 Settings（设置管理）

  • 功能描述：插件设置管理
  • 位置：src/features/settings/
  • 状态：✅ 已实现

3. 技术架构

3.1 分层结构

src/
├── main.ts              # 插件入口（生命周期、DI 容器、命令注册）
├── platform/            # Obsidian API 适配层
│   ├── Vault 端口实现
│   ├── UI 端口实现
│   ├── Modal 端口实现
│   ├── Events 端口实现
│   └── Metadata 端口实现
├── core/                # 核心领域能力
│   ├── settings/        # 设置管理
│   ├── storage/         # 存储抽象
│   ├── ports/           # 端口定义
│   └── utils/           # 核心工具（parser 等）
├── features/            # 功能模块
│   ├── timer/
│   ├── quickinput/
│   ├── aiinput/
│   ├── aichat/
│   └── settings/
├── shared/              # 跨模块共享
│   ├── public/          # 公共 API
│   └── private/         # 内部工具
└── app/                 # 应用装配与运行时
    ├── ServiceManager   # 服务生命周期管理
    ├── usecases/        # 用例编排
    ├── capabilities/    # 能力注册
    └── FeatureLoader    # 功能加载器

3.2 核心设计模式

  • 依赖注入（DI）：使用 DI 容器管理服务依赖
  • 端口适配器：通过 src/platform/ 隔离 Obsidian API
  • 服务管理：ServiceManager 统一管理服务生命周期
  • 功能模块化：每个功能独立在 src/features/<name>/
  • 边界约束：通过 gate 脚本强制架构边界

3.3 工程能力

质量保障

  • Lint：代码风格检查
  • Typecheck：TypeScript 类型检查
  • Test：Jest 单元测试框架
  • Gate：架构边界门禁（多个 gate 脚本）

常用命令

npm run check      # 质量门禁（lint + typecheck + gate）
npm test           # 运行测试
npm run build      # 构建产物

架构门禁（Gate）

位于 scripts/*gate*.mjs，包括：

  • obsidian-leak:gate - 防止平台 API 泄漏到业务层
  • core-obsidian:gate - core 层不依赖 obsidian
  • core-public:gate - core 层 public API 约束
  • shared-public:gate - shared 层 public API 约束
  • iconaction:gate - IconAction 组件使用规范
  • di-resolve:gate - DI 容器使用约束
  • events-boundary:gate - 事件边界约束
  • check-boundaries - 深导入边界检查

L3 领域模型 & 解析层（Core Types + Utils） 定义「世界长什么样」以及“从文本 → 模型”的规则 - src/core/types/schema.ts（Item, FilterRule, SortRule 等结构）
- src/core/utils/parser.ts（任务/块解析：Tags、KV、开始结束时间、时长等）

表 2：数据提取链路（从 Markdown 到“给仪表盘用的数据”）

这一张表就是回答：“从底层 Markdown 扫描 → 成为你仪表盘可用的数据” 的完整路径。

步骤（从下到上） 发生在哪一层 / 文件 输入 处理 输出
1. 读取 Markdown L6 存储 + L5 平台：ObsidianVaultPort Obsidian Vault 的 .md 文件 通过 Obsidian API 读取文件内容；DataStore 只拿到“文件路径 + 文本内容”。 文本字符串（整份 Markdown）
2. 扫描文件 & 解析任务/块 L2 DataStore：src/core/services/DataStore.ts (1) 的 Markdown 文本 + 文件路径/元数据 scanAll / scanFile：
1. 按行遍历，调用 parseTaskLine / parseBlockContent（来自 parser.ts）。
2. 组合 Item：包含文本、状态、标签、时间（start/end/duration）、所属文件、路径等。
3. 结果写入 this.items 和 fileIndex，并更新缓存。		 内存中的 Item[]，按文件分组索引（fileIndex: Map<filePath, Item[]>）。
3. 归一化 & 衍生字段 L3 工具：normalizeItemDates, parseRecurrence, itemFilter Item[] - 把日期/时间字符串标准化成可比较的形式；
- 识别重复任务 / 分类；
- 为后续过滤和排序准备字段。		 带有标准化时间等字段的 Item[]。
4. 条件查询（过滤 + 排序） L2 DataStore：queryItems(filters, sortRules) 全量 Item[] + 上层传入的 FilterRule[] / SortRule[] - 构造 query key，命中则直接用 queryCache。
- 否则用 filterByRules 过滤，再用 sortItems 排序。		 满足当前“视图 / 仪表盘条件”的 Item[] 子集，按排序规则有序。
5. 应用用例包装成“视图数据” L1 App UseCases + Store：例如 blocks.usecase.ts（未来编辑）、视图相关 usecase Item[] + 当前 settings（视图配置、过滤规则等） - UseCase 组合 DataStore 的结果和 Settings；
