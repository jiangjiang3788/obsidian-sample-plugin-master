继续把 features/ 看懂：它像店里顾客真正接触到的服务区

这次只讲 features/。如果说 core 是后仓，app 是店长办公室，那 features 就是顾客真正会走进去、会点击、会使用的那些服务区。你以后整理文件夹时，最关键的问题就是：这个东西是后仓支持？还是顾客直接接触的服务？ 如果是后者，大概率就该进 features。

features/ 导览

先从左边建立感觉：这些都属于“顾客直接接触的服务区”，只是服务内容不同。
  • features/ 总体像前台服务区总览：顾客能看到的功能都在这里
  • aichat/像 AI 咨询台：顾客直接和 AI 聊天的地方
  • aiinput/像智能填单台：顾客输入后交给 AI 处理
  • quickinput/像快速登记台：追求少点几步、马上录入
  • settings/像店里总设置柜台：顾客和管理员都可能来调规则
  • timer/像计时服务区：和时间流逝有关的功能在这里
  • progression/像进度看板区：用来看当前推进到哪

先记这 4 句

  • features 是顾客直接接触到的服务区。
  • features 自己不应该乱改底层状态。
  • features 读状态可以，写动作最好走 usecases。
  • settings 往往最容易长太大，最值得优先整理。

features/ 总体

src/features/
  • 像什么像店里的前台服务区总览。
  • 一句话这里放的是顾客真正能看到、能点击、能直接使用的功能。
  • 它在哪层比 core 更靠前、最接近顾客，属于前台功能层。
  • 主要干什么把聊天、输入、设置、计时、进度这些具体服务区组织起来。
  • 和 core 的区别features 是顾客正在使用的区，core 是后面默默支撑它们的基础支持区。
  • 你怎么记看到 features，就想成“顾客眼前的服务区”。
你整理目录时，先问自己：这个东西是不是顾客直接接触的服务？ 是的话，大概率归 features。

aichat/

src/features/aichat/
  • 像什么像 AI 咨询台。
  • 一句话顾客真正看到并操作的聊天界面、消息区、AI 互动体验在这里。
  • 它在哪层属于前台功能层，是 AI 能力面向顾客的展示区。
  • 主要干什么显示聊天窗口、消息列表、输入互动、聊天相关 UI 流程。
  • 和 core/ai 的区别aichat 管“顾客看到的聊天体验”，core/ai 管“AI 能力本身怎么工作”。
  • 你怎么记看到 aichat，就想成“AI 咨询台本身”。
一句话记忆：AI 的脑子在 core，AI 的窗口在 features/aichat。

aiinput/

src/features/aiinput/
  • 像什么像智能填单台。
  • 一句话顾客输入一点内容，后面由 AI 帮忙理解、补全、处理。
  • 它在哪层属于前台功能层，但更偏“输入交互”而不是完整聊天体验。
  • 主要干什么处理输入框、命令式输入、AI 帮助输入、自动补全类体验。
  • 和 aichat 的区别aiinput 更像“输入时有 AI 帮你”，aichat 更像“你专门打开一个 AI 聊天区去聊”。
  • 你怎么记看到 aiinput，就想成“输入台后面站着个 AI 帮手”。
它和 aichat 容易混，但你记住：一个偏输入过程，一个偏完整聊天场景。

quickinput/

src/features/quickinput/
  • 像什么像快速登记台。
  • 一句话它追求的是少点几步、马上录入，不一定有很复杂的 AI 参与。
  • 它在哪层属于前台功能层，偏高频、快速、轻量操作。
  • 主要干什么快速记录、快速创建、快速填写、尽量缩短操作路径。
  • 和 aiinput 的区别quickinput 强调“快”，aiinput 强调“AI 帮你处理输入”。
  • 你怎么记看到 quickinput，就想成“赶时间时用的快速登记台”。
如果一个功能重点是“少步骤”，不是“更智能”，那它更像 quickinput。

settings/

src/features/settings/
  • 像什么像店里的总设置柜台。
  • 一句话凡是顾客或管理员需要改规则、改显示、改行为的入口，通常都在这里。
  • 它在哪层属于前台功能层，但往往体量最大，容易长成一个子系统。
  • 主要干什么处理主题设置、布局设置、显示规则、编辑器设置、系统行为配置等。
  • 和 core/config 的区别settings 是“前台给人调的设置入口”，core/config 是“系统底层默认配置和规则”。
  • 你怎么记看到 settings，就想成“顾客和管理员都可能会来的设置柜台”。
这通常是最该优先整理的目录。因为它很容易什么都收，最后变成一个大杂烩。

timer/

src/features/timer/
  • 像什么像计时服务区。
  • 一句话和时间流逝、计时展示、时间相关操作有关的顾客功能都在这里。
  • 它在哪层属于前台功能层，是把时间能力包装成顾客能用的功能区。
  • 主要干什么显示计时器、控制开始暂停、展示时间状态、处理时间相关交互。
  • 和 core 里的时间服务区别timer 是顾客操作的计时界面，底层真正记时间、存状态的能力可能在 core/services。
  • 你怎么记看到 timer，就想成“顾客看的秒表区”。
你可以继续沿着这个思路理解：前台展示在 features，后面的持续能力在 core。

progression/

src/features/progression/
  • 像什么像进度看板区。
  • 一句话它负责让顾客看到：现在推进到哪、完成了多少、下一步大概是什么。
  • 它在哪层属于前台功能层，偏展示和反馈类功能区。
  • 主要干什么展示进度、状态趋势、完成情况、推进反馈等界面内容。
  • 和 timer 的区别progression 更关心“做到哪了”，timer 更关心“时间过了多久”。
  • 你怎么记看到 progression，就想成“墙上的进度看板”。
如果一个功能强调的是“进展反馈”，不是“设置”也不是“输入”，那它更像 progression。
