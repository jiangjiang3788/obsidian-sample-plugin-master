左边看“它像店里的什么地方”，右边看“它到底干嘛”

这版不再用一堆积木块解释。左边是通俗导览列表，右边是列表式说明。每个文件夹都先用比喻帮你建立感觉，再用很短的话讲清楚：它在哪一层、干什么、和谁关系最近。

项目导览

看左边先建立感觉：它像什么。再看右边理解它具体做什么。
  • main.ts像商店的开门键：先把整个店开起来
  • app/像店长办公室：负责安排整个店怎么运转
  • core/像后仓和基础设备区：平时看不到，但整个店都靠它
  • features/像店里的各个服务区：顾客直接接触到
  • platform/像店和商场物业的对接窗口：负责和外部环境打交道
  • shared/像店里的公共工具柜：多个区域都能拿来用

先记这 4 句

  • main 负责开店，不负责具体营业。
  • app 像店长办公室，负责安排，不是顾客直接看到的服务区。
  • core 像后仓和基础设备，提供支持，不直接面对顾客。
  • features 是顾客真正接触到的服务区。

main.ts

src/main.ts
  • 像什么像商店的开门键，先把整个店开起来。
  • 一句话它是入口，负责开店，不负责后面的具体营业安排。
  • 层级最外层，排在最前面。
  • 主要作用把店开起来，然后把后面的营业安排交给 app/。
  • 你怎么记看到 main，就理解成“先开店”。
它和 app 的区别：main 是先把店门打开，app 是开门后开始安排今天怎么营业。

app/

src/app/
  • 像什么像店长办公室。
  • 一句话它负责安排：谁先开工、谁来记账、谁来处理具体事务。
  • 层级比 main 更深一层，属于“开店后开始安排营业”的那层。
  • 主要作用把 core 这套后仓能力接起来，再分配给 features 这些前台服务区使用。
  • 里面常见内容store、usecases、bootstrap。你可以理解成：状态、操作入口、启动流程。
  • 你怎么记看到 app，就想成“店长办公室”。
它和 core 的区别：app 负责安排店怎么运转，core 负责提供店运转需要的基础能力。

core/

src/core/
  • 像什么像后仓、库房、基础设备区。
  • 一句话它提供基础支持，但顾客平时不会直接看到它。
  • 层级属于底层，像商店后面的后仓，前台很多地方都要靠它。
  • 主要作用负责数据、设置、AI、工具这些基础支持，就像店里的库房、水电、基础设备。
  • 你最该懂的点它不是给顾客看的服务区，它更像“后仓和基础设备区”。
  • 你怎么记看到 core，就想成“后仓在后面默默撑着整个店”。
你之前看不懂 core，核心原因就是它太抽象。以后直接把它翻成：后仓和基础支持区。

features/

src/features/
  • 像什么像店里的不同服务区：收银区、咨询区、展示区。
  • 一句话这里放顾客真正看得到、点得到、用得到的服务。
  • 层级比 core 更靠前，最接近顾客。
  • 主要作用把聊天、设置、计时器、快速输入这些具体服务区组织起来。
  • 和 app 的关系features 不自己乱改数据，而是找 app 提供的 usecases 去操作。
  • 你怎么记看到 features，就想成“顾客正在使用的服务区”。
如果你在整理文件夹时犹豫一个东西放哪，先问自己：这是顾客直接接触到的服务吗？ 是的话，大概率归 features。

platform/

src/platform/
  • 像什么像店和商场物业的对接窗口。
  • 一句话负责和商场外部环境打交道，再把外部规则接进店里。
  • 层级靠近外部环境，属于“店里和商场外部之间”的一层。
  • 主要作用处理 Vault、Metadata、Events、Modal 这些和外部平台打交道的能力。
  • 你最该懂的点它不是店里的服务本身，它只是负责把店和外部环境接起来。
  • 你怎么记看到 platform，就想成“对外对接窗口”。
如果某段代码里 Obsidian 味很重，大概率就不该散在服务区里，而应该收进 platform 这个对外对接窗口。

shared/

src/shared/
  • 像什么像店里的公共工具柜，多个区域都能来拿。
  • 一句话这里放多个服务区都会用到的公共工具和小零件。
  • 层级不是营业主线的一站，而是横向给各个区域提供支持。
  • 主要作用减少重复，把通用的小工具、小零件统一放一起。
  • 你最该懂的点只有“多个服务区都要用”的东西才该进 shared。
  • 你怎么记看到 shared，就想成“公共工具柜”。
如果一个东西只服务某一个服务区，就别放 shared，放回它自己的功能目录更清楚。
