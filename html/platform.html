继续把 platform/ 看懂：它像店和外部环境的对接窗口

这次只讲 platform/。你可以把它理解成：店里和商场物业、门禁、电表、广播系统打交道的对接窗口。它不是顾客服务区，也不是后仓基础能力本身，而是负责把 Obsidian 这种外部平台能力接进店里。

platform/ 导览

先从左边建立感觉：platform 不是业务区，它是“店内系统”和“外部平台”之间的对接窗口。
  • platform/ 总体像对外对接窗口：负责和 Obsidian 打交道
  • Obsidian*Port.ts像具体窗口员工：按标准把外部能力接进来
  • events/像门口感应器：外面一有变化就通知店里
  • modals/像平台弹窗窗口：和 Obsidian 弹窗体系对接
  • SettingsTab.tsx像物业给的设置入口：挂到平台的官方设置区里
  • platform 的边界像窗口工作守则：只负责对接，不负责业务决策

先记这 4 句

  • platform 负责接外部，不负责决定业务规则。
  • ports 是标准，platform 是按标准接进来。
  • view/UI 如果是顾客直接用的，更偏 features。
  • 平台味很重 的代码，优先收进 platform。

platform/ 总体

src/platform/
  • 像什么像店和外部环境的对接窗口。
  • 一句话它负责把 Obsidian 这种外部平台能力接进项目内部，不直接做顾客业务。
  • 它在哪层在系统边缘，靠近外部平台，属于“对外对接层”。
  • 主要干什么对接 Vault、Metadata、Events、Modal、平台设置页这些外部能力。
  • 和 core/ports 的区别core/ports 是先写“标准”，platform 是具体按这个标准把 Obsidian 接进来。
  • 你怎么记看到 platform，就想成“店里和物业/外部系统打交道的窗口”。
如果一段代码强烈依赖 Obsidian API，那它大概率更像 platform，而不是 features 或 shared。

Obsidian*Port.ts

src/platform/Obsidian*.ts
  • 像什么像具体窗口员工。
  • 一句话这些文件按项目内部约定，把 Vault、Modal、Metadata 这些外部能力真正接进来。
  • 它在哪层属于 platform 里最核心的一层，是“对接的实际实现”。
  • 主要干什么实现 ObsidianVaultPort、ObsidianModalPort、ObsidianMetadataPort 等具体接入逻辑。
  • 和 core/services 的区别platform 负责“把能力接进来”，core/services 负责“拿到能力后持续做事”。
  • 你怎么记看到这些文件，就想成“窗口前的工作人员，按标准处理外部接入”。
这里是“实现层”，不是“定义层”。所以它们和 core/ports 刚好是一组对应关系。

events/

src/platform/events/
  • 像什么像门口感应器。
  • 一句话外面一有变化，比如文件变了、库里更新了，它就把消息带进店里。
  • 它在哪层属于 platform 里偏“监听外部变化”的一层。
  • 主要干什么监听 Vault 变化、文件变化、平台事件，再通知系统内部。
  • 和 app/runtime 的区别events 更像“监听外面发生了什么”，runtime 更像“店开着以后内部怎么持续运转”。
  • 你怎么记看到 events，就想成“门口传感器”。
如果某段代码的重点是“收到外部通知”，那它更像 events，不像 feature 本身。

modals/

src/platform/modals/
  • 像什么像平台弹窗窗口。
  • 一句话这些是直接依赖 Obsidian 弹窗体系的窗口实现。
  • 它在哪层属于 platform 里偏“平台 UI 接入”的一层。
  • 主要干什么实现 QuickInputModal、AiChatModal、NamePromptModal 这类平台级弹窗。
  • 和 features 的区别platform/modals 更像“依赖平台弹窗机制的壳”，features 更像“顾客实际使用的功能内容”。
  • 你怎么记看到 platform/modals，就想成“挂在平台系统上的窗口壳”。
你这个项目里这里和 app/ui/modals、features/quickinput 确实有交叉感，后面很值得整理。

SettingsTab.tsx

src/platform/SettingsTab.tsx
  • 像什么像平台官方设置区里的入口牌。
  • 一句话它负责把你的设置界面挂到 Obsidian 提供的插件设置区里。
  • 它在哪层属于 platform，因为它直接依赖平台给你的设置入口机制。
  • 主要干什么承接平台设置页入口，再转给项目自己的设置内容。
  • 和 features/settings 的区别platform/SettingsTab 管“挂到平台哪”，features/settings 管“里面具体给人改什么”。
  • 你怎么记看到它，就想成“官方设置入口壳”。
这就回答了你“view 是 ui 还是 features”的一部分：挂载到平台哪 更像 platform，里面给人操作什么 更像 features。

platform 的边界

整理时最重要的一页
  • platform 应该放什么强依赖 Obsidian API、平台弹窗、平台设置页、文件系统监听、元数据读取这类代码。
  • platform 不该放什么纯业务规则、顾客直接使用的功能流程、和平台无关的通用组件。
  • view 是 ui 还是 features如果是“系统壳层/挂载层/平台窗口”，更像 ui 或 platform；如果是“某个具体服务区给顾客用的页面”，更像 features。
  • 你该怎么整理把“平台壳”和“功能内容”拆开：壳留在 platform/app/ui，内容尽量回到对应 feature。
  • 一句话总原则platform 负责接进来，不负责决定顾客看到什么业务流程。
你这个项目现在最适合做的，不是继续堆更多层，而是把“平台壳”和“功能内容”剥离清楚。
