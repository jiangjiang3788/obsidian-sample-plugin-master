# 架构补充维度：容易被忽略但对长期很重要的点
日期：2026-02-28

> 背景：  
> 核心架构方向（shared 去平台化 / DataStore 拆分 / 副作用管理 / 竞态与取消）整体是对的，  
> 但要让项目“长久不翻车”，除了这些，还需要关注几个常被忽略的维度。  
> 这份文档就是这些“容易被忘的点”的清单。

---

## 0. 已确认的核心前提（只做复盘，不展开）

1. **Markdown 是唯一事实来源**  
2. **数据源优先级：任务行 + KV 块（callout/thinktxt）**，后续再扩展 frontmatter  
3. **写回必须原位写回**（不搬家、不乱改格式）  
4. **shared 去平台化**（核心/共享层不直接 import obsidian，而是通过 ports/services）  
5. **DataStore 最终需要拆分职责**（解析 / 缓存 / 索引 / 查询）  
6. **副作用统一管理**（事件、定时器、view、modal 都要可 dispose）  
7. **竞态 & 取消有统一策略**（takeLatest / Abort / 可取消 usecase）  

---

## 1. 可观察性（Observability）：日志、调试、问题重现

### 1.1 为什么重要
- 将来你或 AI 排错时，不能只靠“感觉不对”；需要有**足够的信息记录**。  
- 写回 Markdown 出现奇怪结果、AI 行为不对、性能突然变慢，都需要可回溯。

### 1.2 建议要做的几件事
- **统一日志入口**  
  - 提供一个统一的 log 接口（带等级：`debug/info/warn/error`）。  
  - 后续所有模块都通过这个接口打点，而不是随便 `console.log`。  
- **日志要带上下文**  
  - 每条关键日志包含：feature 名、文件路径、itemId、操作类型（scan/write/query…）。  
  - 这样遇到问题时，可以迅速知道“是谁对哪个文件做了什么”。  
- **写回 Markdown 的变更记录**  
  - 对每次写回，记录：文件、位置、旧文本/新文本、调用的 usecase。  
  - 不一定一开始就做得很重，但要预留这个能力。  

> 目标：**出现异常时，你不用“盲猜”，而是可以从日志里还原出“发生了什么”。**

---

## 2. 错误处理与用户提示（Error Handling & UX）

### 2.1 现状风险
- 如果 AI 请求失败、索引重建中断、写回定位失效，目前只是“功能失败”，用户可能不知道发生了什么。  
- 你自己也会因为“看不到清晰错误”而被迫去翻代码。  

### 2.2 建议的方向
- **错误分层**  
  - 业务可预期错误：比如“找不到对应 item”“写回定位失效”“配置不完整”。  
  - 技术错误：网络错误、IO 错误、JSON 解析失败等。  
- **为高频错误写“固定文案 + 操作建议”**  
  - 例如：  
    - 写回失败 → 提示“文件已变化，无法自动写回”，并建议“重新扫描”。  
    - AI 请求失败 → 提示检查网络/API key，并不要丢弃已经输入的内容。  
- **错误提示也走 UiPort**  
  - shared/core 不直接 `new Notice()`，而是通过 UiPort 触发统一的错误展示逻辑。  

> 目标：**出错时，用户知道发生了什么，也知道下一步该做什么，而不是一脸懵。**

---

## 3. 数据演进与兼容性（Settings / Cache / Items Schema）

### 3.1 你现在已经在做的
- 有迁移命令：从 heading 把主题写回任务行（说明你意识到“数据 schema 会变”）。  
- DataStore 与 parser 里也有一些关于字段/语义的调整记录。  

### 3.2 需要系统考虑的点
- **Settings schema**  
  - 每次改设置结构时，是否有版本号？  
  - 旧版本升级时，如何迁移字段/填充默认值？  
- **Cache schema**  
  - 缓存文件是否带版本 / hash？  
  - 不兼容时，是否自动丢弃 cache、重建而不是报错？  
- **Items schema**  
  - 当你给 item 新增字段或改字段名时：  
    - 旧 cache 如何处理？  
    - 旧逻辑/旧测试是否会被打爆？  

> 目标：**DataStore/Settings/Cache 的变更要有“演进故事”，而不是到处临时打补丁。**

---

## 4. 测试策略与测试数据（不是“有没有 Jest”，而是“测什么”）

### 4.1 单测存在不代表“测得对”
你现在已经有 Jest、一些 unit test（例如 legacyCalloutBlock），这是基础。但更重要的是：  

> 哪些链路是必须用测试锁死行为的？  

### 4.2 对 ThinkOS 特别重要的测试点
优先级最高的建议锁定这几块：

1. **parser：Markdown → items**  
   - 给定一段任务行、KV 块、callout，解析出的 item 要符合预期（id/source/filePath/fields/status）。  
2. **patcher：items 操作 → Markdown 文本**  
   - 给定一个 item + 操作（例如勾选完成/更新时间字段），生成的文本 patch 要正确。  
3. **DataStore 增量更新**  
   - 修改单个文件时，DataStore 只更新该文件相关的 items，不破坏其他文件的 items。  
4. **thinktxt 解析规则**  
   - 不同组合的 KV 块 / 边界标记，要有清晰的表格输出测试。  
5. **计时写回**  
   - 给定开始/结束时间，写回出的 `duration` / `time` / `start/end` 字段要稳定。  

> 目标：**只要这些测试是绿的，你就可以放心改架构，而不用担心“行为被悄悄改坏了”。**

---

## 5. 性能与大库策略（渐进扫描 / 分级刷新）

### 5.1 你现在已经有的
- 性能报告命令（尽管你打算后续删掉，但说明你关心性能）。  
- cache + warm start。  
- 重建索引命令。  

### 5.2 可以提前想的策略
- **初次扫描策略**  
  - 大 Vault（很多文件）时：是否考虑“先扫用户最近用的范围，再后台扫描全局”？  
  - 比如优先：当前 workspace 打开的文件 / pinned / starred 路径。  
- **增量更新策略**  
  - 文件内容改变时，避免全库重扫。  
  - DataStore 支持“按文件重建 items”，其余文件不动。  
- **视图局部刷新**  
  - Dashboard 中某个 view 变动时，只刷新与之相关的 items 区域，而不是全量重新渲染。  

> 目标：**在库很大的时候，用户仍然感觉是“渐进可用”的，而不是“一直卡着等扫完”。**

---

## 6. 对外 API / 可扩展性（未来可以先记在脑子里）

### 6.1 问题不是“现在要不要做”，而是“以后有没有这个野心”
要考虑的不是“立刻开放 API”，而是：

- 你是否有可能希望：  
  - 让其他插件/脚本读取 ThinkOS 的 items（查询任务、读取布局配置、访问 Timer 数据）？  
  - 提供命令/API，让别人可以“向 ThinkOS 提交一条记录、触发 Dashboard 更新”？  

### 6.2 如果答案是“有可能”
那就意味着：

- 现在设计 DataStore、UseCase、Feature 对外接口时，要稍微留一点余地。  
- 不要把所有逻辑都写死在 UI 组件或 command 回调里。  

> 目标：**不一定要现在就开放，但不能把将来“开放”的道路堵死。**

---

## 7. 作为“改动前检查清单”怎么用

以后你在想做大改动，例如：

- 拆 DataStore  
- 改写回逻辑  
- 给 thinktxt 加编辑/表格写回  

可以在动手前快速过一遍：

1. shared 去平台化 / DataStore 拆分 / 副作用 / 竞态 ——（你已经在想的）  
2. **可观察性**：这次改动要不要打更多日志？要不要记录写回？  
3. **错误处理**：这次改动失败时，用户会看到什么？  
4. **数据演进**：Settings/Cache/Items 的 schema 是否需要迁移？  
5. **测试**：这次改动是否值得加 1~2 个单测来“锁行为”？  
6. **性能**：对大库会不会更慢？有没有渐进/增量方案？  
7. **扩展性**：这次改动会不会把未来可能的 API 路堵死？  

只要大改动前能在脑子里快速过一遍这几条，  
就算将来某些实现要重写，**方向一般也不会推翻重来**。
