行，那我给你一个**固定可复用的“改代码流程”**，你以后动任何一块，都照这个来就行。
不让你填表，不让你想，只要选「我要改的是哪一类代码」→ 套流程。

---

## 一句话版本

> **先锁住行为 → 再写/修测试 → 再改实现 → 再跑回归**
> 每次改分 8 步，E2E 永远在最后兜底。

---

## 0. 先认清：你这仓库里“代码类型”就 4 种

后面流程里的“分支”只看这 4 类：

| 类型                                         | 你这里的典型文件                                                                    |
| ------------------------------------------ | --------------------------------------------------------------------------- |
| A. 解析/数据层：parser / DataStore / patcher     | `src/core/utils/parser.ts` / `src/core/services/DataStore.ts` / 未来的 patcher |
| B. 设置层：Settings / Layout / ViewModel       | `src/core/services/SettingsRepository.ts` / `src/features/settings/**`      |
| C. UI / 交互层：表格、弹窗、FilterShell 等            | `src/app/ui/**` / `src/features/**` / 未来的编辑表格、dnd                           |
| D. 工程治理层：gate / lifecycle / ServiceManager | `scripts/*gate*.mjs` / `ServiceManager` / `Disposables` / `src/main.ts`     |

**选择你这次要动的主要类型 → 按下面流程走。**

## 改代码流程：统一 8 步

### Step 1：开工前跑一遍“基线”

确保现在的地面是平的，不是你自己踩爆的。

```bash
npm run typecheck
npm run test:unit
# 如果要动 gate / 生命周期，顺带：
npm run gate
```

* 有红：先记下来是哪一块（比如现在的 `legacyCalloutBlock.test.ts`），**不要先修它**，除非你这次任务就是修它。
* 全绿：这次改完如果又红了，就知道是你这次的锅。

---

### Step 2：**写清“我要保证的行为”**（一句话 + 输入输出）

不要写长文档，就一句话 + 一个例子，写在哪里都行（TODO 注释、自己的 md、直接写到测试里都行）。

按类型不同，推荐这么写：

* **改解析/数据层（A）**
  例：

  > “给定含 `[!thinktxt]` 的 callout，`DataStore` 里 `item.content` 必须包含 `- [x]` 任务行。”

* **改设置层（B）**
  例：

  > “在 Settings 里把 `onlyTag = #work` 打开后，仪表盘只显示带 #work 的项。”

* **改 UI/交互层（C）**
  例：

  > “在仪表盘里改一行 title，回车后不跳行，只更新这一行。”

* **改工程治理/gate 层（D）**
  例：

  > “任何 `obsidian` import 只能出现在 `platform/*`，`shared/*` 绝对不允许。”

这一句，就是这次改动的“裁判话术”，后面所有测试/实现都围着它转。

---

### Step 3：先看有没有“现成测试锚点”

**优先复用/修现有测试，不要一上来新建。**

* **解析/数据层（A）**

  * 先搜：`test/unit/legacyCalloutBlock.test.ts`、`parser`、`DataStore` 相关的 test。
  * 现在已经有一个：`legacyCalloutBlock.test.ts` → 这就是你 Phase2 P0 的锚点。

* **设置层（B）**

  * 看 `test/unit/` 里有没有 settings/usecase 相关测试（目前可能较少）。
  * 现有不够，就准备新建一两个“小而精”的。

* **UI 层（C）**

  * 现在 UI 单测不多，可以直接计划未来写 E2E/集成为主。

* **治理/gate 层（D）**

  * 不写 Jest → 全靠 `npm run gate`、`obsidian-leak:gate` 这类脚本。

> 找到了现成测试：**优先让它绿**
> 没有：再按下面一条规则补“最少的测试”。

---

### Step 4：补“最少但关键”的测试（先 test，后实现）

按类型给你钉死：

#### A. 解析 / DataStore / patcher

**目标：金样本，不多写。**

1. 在 `test/unit/` 新建或扩展：

   * `legacyCalloutBlock.test.ts`（现在已经有）；
   * 未来 patcher：`thinktxtPatcher.test.ts`（你可以这么命名）。
2. 每个用例都长这样：

```ts
it('xxx 场景的读→改→写', () => {
  const before = `...markdown...`;
  const ops = [/* update/insert/reorder */];

  const rows = extract(before);      // parser + DataStore
  const after = applyOps(before, ops); // patcher

  expect(after).toBe(`...预期的 md...`);
});
```

**不要碰 UI**，只看字符串。

#### B. Settings / Layout

1. 在 `test/unit/settingsRepository.test.ts`（或类似名字）：

   * 用 `FakeStorage` 初始化 `SettingsRepository`；
   * 测 `load → update → save → reload` 一条闭环。
2. 不要一来就写一堆 UI 设置测试。

#### C. UI / 交互（仪表盘、表格编辑）

1. 对真正“仪表盘编辑体验”，**留给 E2E**（下面 Step 7 说）。
2. 组件级测试只写：

   * Shell 是否渲染；
   * 点击/拖拽能不能触发某个 usecase 调用（至多一两条）。

#### D. gate / lifecycle

* 全交给脚本：

  * `npm run obsidian-leak:gate`
  * `npm run check:boundaries`
  * `npm run gate`

---

### Step 5：再动实现（只动必要的层）

开始写/改代码时，按“**尽量往下动，不往上乱牵**”的顺序：

1. **优先改 A 层（解析/DataStore/patcher）**

   * `parser.ts`、`DataStore.ts`、未来的 patcher。
   * 不要一上来动 UI；先保证数据层行为对。
2. 然后视情况改 B 层（Settings / 用例）

   * 如果这次行为变更需要新的设置/过滤规则，再在 `SettingsRepository` 和 usecases 里补。
3. UI 层（C）最后再改、越薄越好

   * 只是绑定事件、展示数据，尽量不放业务逻辑。
4. 工程/gate 层（D）

   * 只有当你在做 Phase0/1 那几项工程化任务时才动它。

> 简单讲：**行为规则写在 Core（A/B），UI 做一层壳，Platform/gate 不进业务逻辑。**

---

### Step 6：跑“和这次改动匹配”的测试组合

这里直接给你一个表，按「改的类型」选命令：

| 这次主改动类型                | 必跑命令                                      | 推荐多跑一条                            |
| ---------------------- | ----------------------------------------- | --------------------------------- |
| A 解析/DataStore/patcher | `npm run typecheck` + `npm run test:unit` | 如果动 DataStore：再跑 1 条相关集成测试（你之后会有） |
| B Settings             | 同上                                        | 选 E2E-4（“改设置 → 仪表盘变化”）跑一次         |
| C UI/交互                | `npm run typecheck` + 相关 UI/用例的 unit/集成   | 遍历相关 E2E 场景（比如“编辑 title 写回”）      |
| D gate/lifecycle       | `npm run typecheck` + `npm run gate`      | 生命周期改动时加：E2E-5（reload 后再编辑）       |

你在意 E2E，就给几条场景编号，之后固定用：

* E2E-1：预览表格能出来（只读）
* E2E-2：编辑一行写回成功
* E2E-3：拖拽排序写回成功
* E2E-4：改设置 → 仪表盘变化
* E2E-5：reload/unload 后没有鬼畜行为

每次大一点的改动，至少跑和这次相关的 1 条 E2E。

---

### Step 7：如果有 E2E，最后才改它 / 修它

你比较重视端到端，所以单独说一下节奏：

1. **先让单测/集成绿**

   * UT/IT 绿了，说明行为在数据层是稳定的。
2. 再跑 E2E

   * 如果挂了，多半是 UI/绑定/API 名字之类的问题。
3. 修 E2E 的时候，**尽量只动 UI / usecase**，别再往下层继续挖坑。

这样你可以做到：

* patcher / DataStore / Settings 这几块行为一旦定了，后面基本不用改；
* 真正需要跟需求一起变的是 UI 和少量用例层代码。

---

### Step 8：收尾（记录一下这次改动的“锚点”）

最后一分钟做两件小事：

1. 在测试里留一条清楚的描述

   * 比如 UT 用例名：`it('thinktxt callout 中的任务行 content 必须包含原始 - [x] 行')`
   * 将来你再看这个 test file，就知道「这条就是那次我修的行为」。
2. 在任务表/commit message 里写一句：

   * `Fix: Phase2 P0 legacy block -> legacyCalloutBlock.test.ts 全绿`

以后你回顾这个 Phase 表，就能直接对上哪条测试在保护哪一块。

## 如果你要一个**极简背诵版**

> 以后每次改东西，脑子里就背这个顺序：

1. **跑基线**：typecheck + test:unit (+ gate)
2. **一句话写清这次要保证的行为**
3. **先找测试锚点，没有就补最少的 UT/IT**
4. **先改 Core（parser/DataStore/Settings），后改 UI**
5. **跑匹配的测试组合：UT → IT → E2E**
6. **最后在测试名+commit 里把行为写死**

照这个节奏，你就是从“盲改”变成“有设计地改”，而且**先把测试和行为想清楚，再去碰代码**，不容易在后面发现“白写了一堆测试”。
