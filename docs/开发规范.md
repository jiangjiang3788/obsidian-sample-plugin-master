## 🎯 新人必读（三分钟）

### 最小必知（🔴 必须）

1. **入口文件**：`src/main.ts`（插件启动点）
2. **构建命令**：`npm run build`（生成 `main.js`/`styles.css`）
3. **检查命令**：`npm run check`（提交前必跑）
4. **架构铁律**（违反会 gate 失败）：
   - ❌ 不能在 `src/platform/` 外直接 `import 'obsidian'`
   - ❌ 不能在 `features/A` 里引用 `features/B`
   - ❌ 不能在 feature/shared 里用 `tsyringe`/`container`

### 分层速览（🟡 重要）

```
src/
├── platform/     ← 🔴 唯一能碰 Obsidian API 的地方
├── core/         ← 核心领域（对外用 @core/public）
├── app/          ← 应用装配（ServiceManager/capabilities）
├── features/     ← 业务功能（timer/aiinput/aichat...）
└── shared/       ← 共享 UI/工具（对外用 @shared/public）
```

## 1. 开发流程（建议）

- 先定位入口：
  - 命令/入口：`src/main.ts`
  - 功能模块：`src/features/**`
- 改完必跑（至少其一）：
  - 快速：`npm run lint` + `npm run typecheck`
  - 完整：`npm run check`（lint + typecheck + gate）
- 涉及 UI/样式：跑 `npm run build`，并确认根目录 `main.js/styles.css` 已更新（构建会从 `dist/` 复制到根目录）。
- 验证到 Obsidian：把 `manifest.json/main.js/styles.css` 复制到 Vault 的插件目录并 reload 插件（见第 9 节）。

## 2. 代码风格（最基础）

目标：让代码“看起来统一、可读”，并且尽量让 lint/typecheck 能自动兜底。

> 本仓库目前以 ESLint + TypeScript `strict` 为主，没有强制引入 Prettier；因此最低要求是：遵循现有写法，并通过 `npm run lint`/`npm run typecheck`。

### 2.1 命名规范

- 变量/函数：`camelCase`（例：`userName`）
- 常量：`UPPER_SNAKE_CASE`（例：`MAX_RETRY_COUNT`）
- 类/服务/组件：`PascalCase`（例：`UserService`、`TimerView`）
- 布尔变量：`isXxx`/`hasXxx`/`canXxx`/`shouldXxx`（例：`isEnabled`、`hasPermission`）
- 文件名：跟随仓库现有风格（常见为 `PascalCase.ts(x)` 或 `lowerCamel.suffix.ts`，例如 `TimerService.ts`、`floatingWindows.slice.ts`；避免随意引入 `kebab-case` 新风格）

### 2.2 格式规范

- 缩进：跟随现有文件风格；本仓库大多数 TS/TSX 使用 **4 空格**，不要在同文件混用 tab/空格
- 每行长度：不硬性限制；超长行建议在 100~120 字符附近主动换行（参数列表、对象字面量、长 JSX）
- 大括号/换行：保持同文件一致（同一文件不要出现两套括号风格）
- 空行：禁止“为了好看”堆空行；两段逻辑之间最多保留 1 行空行
- 工具优先级：
  - ESLint：必须通过（`npm run lint`）
  - TypeScript：必须通过（`npm run typecheck`）
  -（如未来引入 Prettier，会在此处补齐配置与运行方式）

### 2.3 注释规范

- 公共 API/跨模块边界（例如 `@core/public`、`@shared/public`、capability 合同）应使用 JSDoc 描述用途、参数与返回值
- 复杂逻辑/非直观代码必须解释“为什么”，不要只描述“做了什么”
- 禁止废话注释（例如 `x = 1` 上写“设置 x 为 1”）

好的注释示例：

```ts
/**
 * 根据用户 ID 获取用户信息（会命中缓存；找不到则返回 null）。
 * @param userId 用户唯一标识
 */
export async function getUserById(userId: string): Promise<User | null> {
  // ...
}
```

### 2.4 console / 日志规范（已由 ESLint 部分约束）

- `src/**/*` 默认 `console` 是 `warn`（允许但提示你别扩散）
- 关键路径（例如 `src/main.ts`、`src/app/usecases/**`、`src/app/store/slices/**`、`src/core/ai/**`）禁止 `console`（`error`，见 `.eslintrc.cjs`）
- 如需日志：优先使用封装（例如 `devLog` 等），不要在业务逻辑里到处散 `console.log`

### 2.5 提交前自检

```bash
npm run lint
npm run typecheck
npm run check
```

## 3. 代码结构与单一职责（SRP）

目标：让项目可维护、可扩展，并且不破坏现有分层与门禁（gate）。

### 3.1 目录结构与分层

遵循本仓库现有分层（简化理解）：

- `src/platform/`：唯一允许直接调用 Obsidian API
- `src/core/`：核心领域与基础设施（对外门面 `@core/public`）
- `src/app/`：应用装配与运行时（ServiceManager/usecases/capabilities/bootstrap）
- `src/features/`：具体功能模块（timer/aiinput/aichat/...）
- `src/shared/`：共享 UI/工具（对外门面 `@shared/public`）

### 3.2 单一职责（SRP）扩展约定

- 一个文件只聚焦一类职责（例如 service / view / selector / slice）；不要把“UI + 数据持久化 + 网络请求 + 状态管理”塞进一个文件
- 一个函数只做一件事：超过 ~50 行且包含多层分支时，优先拆分小函数，并给拆出来的函数起“表达意图”的名字
- 一个类只对一个模块负责：避免“万能 Service”；如果类需要依赖太多东西，通常意味着职责过大
- 边界处（ports/capabilities/public facade）尽量保持“薄”：只做参数校验、适配、编排，不把业务细节塞进边界层

### 3.3 模块依赖规则（用 gate 强制）

- 禁止循环依赖（尤其是 core 内部反向依赖 `@core/public` 会导致 TDZ）
- `features/<A>` 禁止直接依赖 `features/<B>`（用 capability/public/shared 连接）
- 非 `platform` 禁止直接 `import 'obsidian'`（用 ports 间接访问）

## 4. 编码原则与代码质量（建议长期遵守）

### 4.1 编码原则（优先级从高到低）

- 可读性优先：代码首先是写给”未来的自己/同事/AI”看的；宁可多写几行，也不要为省行数牺牲可读性
- 显式依赖：依赖通过参数/构造注入传递；避免在业务代码里”到处 resolve / 到处单例”
- 最小暴露面：跨模块只暴露必要的类型/能力；缺 export 就补到 `@core/public` / `@shared/public`，不要 deep import
- 小步改动：能拆 PR 就拆；避免”重构 + 新功能 + 格式化”混在一起
- 不过度抽象：优先把重复收敛在同一处再抽；遵循 KISS/YAGNI，别为了”可能复用”提前造框架

### 4.1.1 类型安全（TypeScript 规范）

- 禁止滥用 `any`：必须使用时加注释说明原因；优先用 `unknown` + 类型守卫
- 禁止 `as` 强制断言（除非你 100% 确定）：优先用类型守卫（`if (typeof x === 'string')`）
- 禁止 `@ts-ignore` / `@ts-expect-error`（除非是已知的第三方库类型问题）：加注释说明原因和 issue 链接
- 优先使用严格类型：
  - 函数参数/返回值必须显式标注类型（不依赖推断）
  - 公共 API 必须有完整类型定义
  - 避免 `object`/`Function` 等宽泛类型
- 善用工具类型：`Partial<T>`、`Pick<T, K>`、`Omit<T, K>`、`Record<K, V>` 等

示例：

```ts
// ❌ 不好
function process(data: any) {
    return data.value as string;
}

// ✅ 好
function process(data: unknown): string {
    if (typeof data === 'object' && data !== null && 'value' in data) {
        const value = (data as { value: unknown }).value;
        if (typeof value === 'string') {
            return value;
        }
    }
    throw new Error('Invalid data format');
}
```

### 4.1.2 异步操作规范

- 优先使用 `async/await`（比 `.then()` 链更可读）
- 必须处理 Promise rejection：
  - 顶层异步操作必须 `try/catch`
  - 或者用 `.catch()` 兜底
- 避免”悬挂 Promise”：
  - 不要 `void someAsyncFunc()`（除非你确定不需要等待结果）
  - 如果确实是”fire and forget”，加注释说明
- 并发控制：
  - 批量操作用 `Promise.all()`（注意一个失败全失败）
  - 需要容错用 `Promise.allSettled()`
  - 需要竞速用 `Promise.race()`
- 超时保护：关键异步操作（网络/AI）必须设置超时

示例：

```ts
// ❌ 不好：悬挂 Promise
function onClick() {
    fetchData(); // 没有 await，错误会被吞掉
}

// ✅ 好：显式处理
async function onClick() {
    try {
        await fetchData();
    } catch (error) {
        console.error('Failed to fetch:', error);
        // 显示错误提示
    }
}

// ✅ 好：fire and forget 加注释
function onClick() {
    // Fire and forget: 预加载数据，失败不影响当前操作
    void fetchData().catch(err => console.warn('Prefetch failed:', err));
}
```

### 4.1.3 性能规范

- React 渲染优化：
  - 大列表必须虚拟化（react-window / react-virtualized）
  - 避免在 render 里创建新对象/函数（用 `useMemo` / `useCallback`）
  - 避免”整棵 state 订阅”（已有 gate 约束）
- 防抖/节流：
  - 搜索输入：防抖（debounce）300-500ms
  - 滚动/resize 事件：节流（throttle）100-200ms
  - 频繁触发的计算：用 `useMemo` 缓存
- 内存泄漏防护：
  - 组件卸载时必须清理：定时器、订阅、事件监听
  - 大对象用完及时释放引用
  - 避免闭包意外持有大对象
- 避免阻塞主线程：
  - 大量计算考虑 Web Worker（如果适用）
  - 长列表渲染分批处理（`requestIdleCallback` / `setTimeout` 分片）

示例：

```ts
// ❌ 不好：每次渲染创建新函数
function MyComponent({ items }: { items: Item[] }) {
    return items.map(item => (
        <button onClick={() => handleClick(item.id)}>
            {item.name}
        </button>
    ));
}

// ✅ 好：缓存回调
function MyComponent({ items }: { items: Item[] }) {
    const handleClick = useCallback((id: string) => {
        // ...
    }, []);

    return items.map(item => (
        <button onClick={() => handleClick(item.id)}>
            {item.name}
        </button>
    ));
}
```

### 4.2 禁止事项（最小集合）

- 禁止写死配置（路径/开关/端点尽量进入 settings 或集中常量）
- 禁止魔法数字（抽常量；或者用具名配置）
- 禁止复制粘贴式代码扩散（抽到 `shared`/`core`，或用通用函数）
- 禁止在关键路径残留 `console.log`（仓库已有 ESLint 规则 + gate 思路）
- 禁止空 `catch {}`（至少要：说明为什么可以忽略，或记录/上抛）

### 4.3 错误处理与资源释放

- 异步流程要保证资源释放/订阅释放（尤其是与 Obsidian 生命周期相关的对象）
- Promise 型 Modal 必须在 `onClose` 里 resolve（仓库有 gate 兜底，见第 13 节）
- 对外部依赖（文件系统/网络/AI）要有失败兜底：超时、错误提示、可重试路径，不要把异常吞掉

### 4.4 日志分级（建议）

- `INFO`：正常流程关键节点
- `WARN`：可恢复异常/退化路径
- `ERROR`：需要关注的严重问题（并尽量带上下文）

## 5. 测试规范

- 核心纯逻辑必须有单元测试（Jest：`npm test`）
- 新功能至少补一条“最关键路径”的测试（纯逻辑优先；UI/平台侧可用集成/e2e 覆盖）
- 覆盖率：当前仓库未硬性门槛；如果以后要加，建议从核心模块开始设 70% 起步

## 6. 安全与隐私（必须遵守）

- 不提交密钥/Token/个人隐私数据（包括 `.env`、日志 dump、测试账号等）
- settings 持久化必须脱敏：仓库有 `settings-persistence-gate` 明确要求剥离 `aiSettings.apiKey`（见第 13 节）
- 禁止把用户笔记内容/路径/密钥直接打进日志；如需排障，先做脱敏/截断，并避免在默认日志级别输出
- 网络请求/AI 请求必须做输入校验与错误处理（避免把用户笔记内容/路径意外泄露到日志或外部）
- 渲染/拼接 HTML 时避免直接使用不可信输入（避免注入/XSS）；优先使用 Obsidian/项目内既有渲染与转义工具

## 7. 提交规范（非常重要）

目标：避免版本管理混乱，便于回溯与自动生成发布说明。

### 7.1 Commit Message（建议 Conventional Commits）

- `feat: ...` 新功能
- `fix: ...` 修复
- `refactor: ...` 重构（不改外部行为）
- `docs: ...` 文档
- `test: ...` 测试
- `chore: ...` 构建/脚本/杂项

示例：

- `feat(timer): stop and apply active timer`
- `fix(aiinput): sanitize apiKey before persist`

### 7.2 分支规范（建议）

- `main`：稳定分支（不直接提交）
- `feature/<topic>`：功能
- `fix/<topic>`：修复
- 所有改动走 PR + Code Review（即使是你自己，也建议用 PR 当“变更记录”）

## 9. 在 Obsidian 中加载与调试

### 9.1 手动安装/开发加载

最小插件发布物：

- `manifest.json`
- `main.js`
- `styles.css`

复制到：

`<你的Vault>/.obsidian/plugins/think-os/`

PowerShell 示例：

```powershell
$vault = "D:\ObsidianVault"
$pluginDir = Join-Path $vault ".obsidian\plugins\think-os"
New-Item -ItemType Directory -Force -Path $pluginDir | Out-Null
Copy-Item -Force manifest.json, main.js, styles.css $pluginDir
```

### 9.2 调试建议（最常用）

- 打开开发者工具：Obsidian 内置 DevTools（Console/Network/Sources）。
- 日志入口：`src/main.ts` 会输出启动日志（你也可以在关键路径加 `console.log` / 项目内日志工具）。
- 源映射：仓库根目录的 `main.js.map` 用于 DevTools 映射回 `src/**`。
- 常见“改了没生效”原因：
  - 你没有把新的 `main.js/styles.css` 拷贝到 Vault 的插件目录
  - Obsidian 没有 reload 插件 / 没有重启 app
  - Vault 下插件目录里仍是旧文件（路径写错、复制失败）

## 10. 入口与启动顺序（从 main.ts 开始）

- 插件入口：`src/main.ts`
- `onload()` 做的关键事情（按代码语义）：
  - 加载 settings（`loadData()` 合并 `DEFAULT_SETTINGS`）
  - 初始化 core DI（`setupCoreContainer(this.app, settings)`）
  - 注册 platform 端口实现到容器（Vault/UI/Modal/Events/Metadata/FileStat/MessageRender 等 token）
  - 启动 `ServiceManager.bootstrap()`（组装 useCases / services）
  - 构建 capabilities（`createCapabilities(...)` + registry）
  - 注册命令（`registerCommands()`）
  - 注册 thinktxt 预览渲染（`registerThinktxtPreview(this)`）

你要找“功能入口/命令入口”，优先从 `src/main.ts` 的 command 注册和 `src/features/` 对应模块开始。

## 11. 分层与目录职责（速查）

- `src/platform/`：唯一允许直接接触 Obsidian API 的地方（ports/adapters）。
- `src/core/`：核心领域与基础设施（对外门面是 `src/core/public.ts`）。
- `src/app/`：应用装配层（ServiceManager、usecases、capabilities、FeatureLoader、bootstrap）。
- `src/features/`：业务功能模块（timer / quickinput / aiinput / aichat / settings ...）。
- `src/shared/`：共享工具与共享 UI（对外门面是 `src/shared/public.ts`）。

## 12. 开发时最重要的“硬规则”（AI 代写时必须遵守）

### 12.1 Obsidian API 只能在 platform 用

- gate：`scripts/obsidian-leak-gate.mjs`
- 规则：除 `src/platform/**`（+ allowlist）外，禁止任何地方 `import 'obsidian'`
- 修法：
  - 把 `obsidian` 相关调用下沉到 `src/platform/` 的 Port 实现
  - 业务层通过 Port 接口/Token 注入获取能力

### 12.2 features 之间禁止互相引用

- gate：`scripts/feature-gate.mjs`
- 规则：`src/features/<A>` 禁止直接 import `src/features/<B>`（A!=B）
- 修法：
  - 通过 `app/capabilities` 做“协作接线”
  - 或把公共内容抽到 `core`（合同/领域）或 `shared`（UI/工具）

### 12.3 features/shared 禁止直接使用 tsyringe（DI 权力集中）

- gate：`scripts/di-gate.mjs`
- 规则：
  - `src/features/**`、`src/shared/**` 禁止引入 `tsyringe`
  - 禁止 `container.resolve/register/...`、禁止 `@singleton/@inject...`
- 修法：
  - 把装配/resolve 移到 composition root（例如 `src/main.ts` / `src/app/ServiceManager.ts` 等）
  - feature 侧只接收“注入后的能力/服务”（capability / usecase）

### 12.4 container.resolve() 只能出现在极少数根文件

- gate：`scripts/di-resolve-gate.mjs`
- 当前规则：只允许在 `src/app/bootstrap/buildRuntime.ts` 使用 `container.resolve(...)`
- 修法：把 resolve 收敛到 buildRuntime，通过参数传递把依赖传下去

### 12.5 public facade（门面）规则：禁止深导入/禁止循环依赖

- core 对外门面：`@core/public`
  - gate：`scripts/public-api-gate.mjs`（非 core 层禁止 deep import core，只能 `@core/public`）
  - gate：`scripts/core-public-gate.mjs`（core 内部禁止 import `@core/public`，避免循环依赖/TDZ）
- shared 对外门面：`@shared/public`
  - gate：`scripts/shared-public-gate.mjs`（shared 之外禁止 deep import shared，只能 `@shared/public`）

修法要点：

- **core 内部**：直接 import 具体模块路径，不要绕回 `@core/public`
- **core 外部**：尽量只从 `@core/public` 拿需要的类型/能力；如果缺 export，就在门面补 re-export
- **shared 外部**：只从 `@shared/public` 使用 UI/工具；缺 export 就补到 `src/shared/public.ts`

### 12.6 事件订阅只能在 platform（防泄漏/防忘记 dispose）

- gate：`scripts/events-boundary-gate.mjs`
- 规则：除 `src/platform/**` 外，禁止 `app.vault.on` / `app.workspace.on` / `registerEvent(...)`
- 修法：把事件订阅放到 platform 的 `EventsPort` 实现里；业务层只消费 EventsPort

### 12.7 防“新旧两套系统并存/回潮”

- gate：`scripts/dual-system-gate.mjs`
- 规则（文本门禁）：
  - `FeatureLoader` 禁止直接 `registry.register(...)`（注册必须集中化）
  - `src/main.ts` 禁止 `capabilityRegistry.register(...)`
  - `src/app/capabilities/createCapabilities.ts` 禁止 `registry.register(...)`
- 修法：把注册移动到集中注册文件（脚本里提示的路径），让入口文件只做“编排”

## 13. 质量门禁（gate）一览：约束什么、失败怎么修

你跑 `npm run gate` 会依次执行这些脚本（见 `package.json`）。

### 13.1 架构/边界类

- `scripts/arch-gate.mjs`：AST 级别的分层/出口规则（导入解析更可靠）
- `scripts/public-api-gate.mjs`：core/public 只允许从指定 barrel re-export；非 core 禁止 deep import core
- `scripts/feature-gate.mjs`：feature 互相隔离（features/<A> 禁止引用 features/<B>）
- `scripts/di-gate.mjs`：features/shared 禁止使用 tsyringe/container/DI 装饰器
- `scripts/di-resolve-gate.mjs`：container.resolve 只能在 composition root（当前收敛到 buildRuntime）
- `scripts/obsidian-leak-gate.mjs`：除 platform/allowlist 外，禁止 import 'obsidian'
- `scripts/core-obsidian-gate.mjs`：core 内 obsidian 依赖冻结扩散（allowlist 外禁止）
- `scripts/events-boundary-gate.mjs`：事件订阅只允许在 platform
- `scripts/core-public-gate.mjs`：core 内禁止 import `@core/public`（防循环依赖/TDZ）
- `scripts/shared-public-gate.mjs`：shared 外禁止 deep import shared（只能 `@shared/public`）
- `scripts/dual-system-gate.mjs`：防注册分散导致“新旧系统并存/回潮”
- `scripts/capability-gate.mjs`：capabilities 文件导出规则（禁止 export*、禁止导出实现物、限制导出数量）

### 13.2 体验/回归类（针对特定坑的硬闸）

- `scripts/settings-persistence-gate.mjs`：持久化时必须 sanitize，且必须剥离 `aiSettings.apiKey`
- `scripts/modal-promise-gate.mjs`：Promise 型 Modal 必须在 onClose resolve（避免 Promise 悬挂）
- `scripts/selector-giant-subscription-gate.mjs`：禁止对 zustand store 的 “整棵 state/settings” 订阅（避免大范围重渲染）
- `scripts/theme-tree-recursion-gate.mjs`：禁止 ThemeTreeNodeRow 递归渲染自身
- `scripts/theme-matrix-legacy-import-gate.mjs`：禁止 ThemeMatrixView 使用 legacy buildThemeMatrixTree
- `scripts/iconaction-gate.mjs`：禁止回退到 `<Tooltip><IconButton/></Tooltip>` 旧模式（改用 `IconAction`）

## 14. 常见失败案例（快速定位）

- `obsidian-leak-gate` 失败：
  - 你在非 `src/platform/**` 的文件里写了 `import { ... } from 'obsidian'`
  - 修：把这段逻辑下沉到 `src/platform/`，上层通过 Port 调用
- `feature-gate` 失败：
  - `features/A` 直接引用了 `features/B`
  - 修：改为 capability 协作，或抽到 core/shared
- `shared-public-gate` / `public-api-gate` 失败：
  - 你 deep import 了 `@shared/...` 或 `@core/...`
  - 修：改从 `@shared/public` / `@core/public` 导入；缺 export 就补 re-export
- `di-gate` / `di-resolve-gate` 失败：
  - 你在 feature/shared 里用了 tsyringe/container 或在非允许文件 resolve
  - 修：把 resolve 移到 buildRuntime 或 ServiceManager，依赖通过参数/构造注入传递
- `settings-persistence-gate` 失败：
  - 你改了 settings 持久化逻辑但没剥离 `aiSettings.apiKey`
  - 修：确保 `sanitizeForPersistence` 存在并被 `saveData(sanitizeForPersistence(settings))` 调用（见 gate 报错提示的文件）
