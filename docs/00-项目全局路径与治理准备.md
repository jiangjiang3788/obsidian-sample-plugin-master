# 项目全局路径与治理准备（AI 协作版）

> 适用场景：你负责提需求，AI 负责实现；你希望项目按标准流程稳定推进。

## A. 全局路径（先看全局，再做需求）






1. 项目治理准备（规则先行）
2. 需求分析（定义做什么、怎么验收）
3. 方案设计（产品方案 + 技术方案）
4. 迭代计划（拆任务、排优先级、定里程碑）
5. 开发实现（AI 编码 + 本地运行）
6. 质量校验（lint / typecheck / test / gate）
7. 代码评审（变更说明、风险点、回滚点）
8. 发布上线（构建产物、版本记录）
9. 运行监控（错误、性能、用户反馈）
10. 复盘迭代（问题回收进入下轮需求池）

每一轮功能都走这条路径，项目会越来越稳。

## B. 你这个项目的现状（已完成的基础）

已具备较好的工程底座：

- 依赖与构建：`vite` + TypeScript
- 质量命令：`npm run check`、`npm run gate`、`npm run test`
- 架构门禁：`scripts/*gate*.mjs` 已存在

结论：你可以直接进入“项目治理准备”的执行阶段，不需要从零搭架子。

## C. 项目治理准备（从今天开始执行）

### C1. 治理目标（非技术视角）

- 任何功能都可追踪：为什么做、做到哪、何时完成
- 任何改动都可验证：有明确验收标准
- 任何问题都可回溯：知道是谁、何时、改了什么

### C2. 最小治理规则（MVP）

1. **先写需求卡，再让 AI 写代码**（禁止“先改后补”）
2. **一次只做一个功能卡**（减少混乱）
3. **每次合并前跑固定检查**：`npm run check` + `npm run test:unit`
4. **每次功能完成必须回写文档**（状态改为 Done）
5. **高风险改动必须有回滚说明**（至少一句）

### C3. 角色分工（你 vs AI）

- 你负责：目标、优先级、验收标准、是否上线
- AI 负责：方案、编码、测试、改动说明
- 共同负责：把结果写回文档，保持项目可管理

## E. 需求卡模板（你每次直接填这个给 AI）

复制以下模板：

```md
## 功能名

### 1) 背景/目标
- 为什么要做：
- 不做会怎样：

### 2) 用户场景
- 谁在什么场景下需要什么能力：

### 3) 验收标准（必须可测试）
- [ ] 标准1：
- [ ] 标准2：
- [ ] 标准3：

### 4) 范围边界
- 本次要做：
- 本次不做：

### 5) 风险与回滚
- 主要风险：
- 回滚方式：
```

## F. 每次功能执行流程（固定 6 步）

1. 你提交“需求卡”
2. AI 产出“技术任务清单”
3. AI 实现代码并说明改动文件
4. AI 运行/指导运行检查命令
5. 你按验收标准确认结果
6. AI 回写文档与变更记录
