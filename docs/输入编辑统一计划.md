## 四、quickinput 这次到底怎么处理

### 不建议的两种做法

| 方案                    | 为什么不建议                |
| --------------------- | --------------------- |
| 继续直接把 `quickinput` 做大 | 会越来越像“输入系统本体”，名字和职责失真 |
| 再做一个新输入前台入口           | 真的会变成双渠道              |

### 建议的做法

| 做法                                  | 说明                                                      |
| ----------------------------------- | ------------------------------------------------------- |
| 保留 `quickinput` 作为前台入口              | 用户入口不变                                                  |
| 新建统一输入内核                            | 真正的 create / update / delete / validate / normalize 放进去 |
| 让 `quickinput` 先接统一内核               | 先做一条主线                                                  |
| 后面再让 view / modal / settings 入口逐步接入 | 最后所有入口走同一套动作                                            |

### 一句话

**quickinput 留着，但降级成入口壳。**

## 五、推荐的统一动作模型

你先不要做太大，先收成这些标准动作：

| 动作名                          | 作用                     |
| ---------------------------- | ---------------------- |
| `openCreateRecord`           | 打开新建入口                 |
| `openEditRecord`             | 打开编辑入口                 |
| `submitCreateRecord`         | 提交新建                   |
| `submitUpdateRecord`         | 提交修改                   |
| `submitDeleteRecord`         | 提交删除                   |
| `validateRecordInput`        | 校验字段 / 类型 / 主题等是否合法    |
| `normalizeRecordInput`       | 统一补默认值和缺失项             |
| `resolveMissingDependencies` | 统一处理字段缺失 / 类型缺失 / 主题缺失 |

重点是：

**UI 只负责触发这些动作。**
**UI 不自己决定怎么保存、怎么补齐、怎么刷新。**

## 六、副作用管理这次要怎么纳入

这次不要另开一整套大工程，但一定要纳入主计划。

### 你至少要纳入这 4 类副作用

| 副作用类型                                | 要不要这次纳入 |
| ------------------------------------ | ------- |
| 数据写入（store / repository）             | 要       |
| UI 反馈（toast / modal / panel）         | 要       |
| 视图刷新（view / layout / widget refresh） | 要       |
| 防重与竞态（重复提交 / 覆盖更新 / 删除冲突）            | 要       |
| 埋点 / 日志                              | 可以后放    |
| 性能优化                                 | 可以后放    |

### 推荐归属

| 逻辑                            | 放哪                           |
| ----------------------------- | ---------------------------- |
| 点击按钮 / 打开弹窗                   | UI 层                         |
| create / update / delete 主动作  | `app/usecases`               |
| validate / normalize          | `core` 的输入服务或 validation 层   |
| 写 store / repository          | store / repository           |
| toast / refresh / close modal | 尽量由 usecase 统一触发或由调用方按统一结果处理 |

## 七、竞态和防重，这次顺便定掉

这次不一定一次修完，但规则必须先定。

| 问题                               | 最低处理标准                |
| -------------------------------- | --------------------- |
| 连点两次提交                           | 提交中锁按钮                |
| 同一 record 被多个入口同时编辑              | 同 `recordId` 串行提交     |
| 删除和更新同时发生                        | 更新前检查对象仍存在            |
| 缺失补全覆盖用户输入                       | normalize 只在提交前做一次    |
| quickinput 和 view editor 同时改一条记录 | 统一走同一个 update usecase |
| modal 保存后 view 没刷新               | 提交成功后统一刷新路径           |

你满足下面这些，就可以正式动手：

| 标准                                     | 达到说明        |
| -------------------------------------- | ----------- |

| 字段缺失 / 类型缺失 / 主题缺失逻辑已找全                | 校验规则能收口     |
